How to use amq-squad

amq-squad is the project layer above AMQ. It records who is on the team, what role each agent plays, which binary each role runs, and how to launch the whole team back into the same AMQ workstream.

1 shared AMQ session per issue, release, branch, or focused workstream
n threads inside that session for reviews, decisions, and role handoffs
0 required AMQ changes. amq-squad launches through normal AMQ commands

Install

Install the published binary with Go. The command below installs into $GOBIN or $HOME/go/bin. 

go install github.com/omriariav/amq-squad/cmd/amq-squad@v0.5.1
amq-squad version

You also need Go 1.25 or newer, the amq binary in PATH, and tmux if you want automatic pane launch.

Core model

AMQ is the message queue. amq-squad is the team operating layer. It keeps durable team intent in the project and writes per-agent launch records when each role starts.

AMQ owns messaging

AMQ delivers messages, keeps mailboxes, tracks sessions, and exposes commands such as amq send, amq list, and amq drain.

mailbox session thread receipt
amq-squad owns teams

amq-squad stores the roster, role docs, shared rules, member working directories, and exact launch commands needed to recreate the team.

roles binaries team rules launch records

Quick start

Start in the repository that should own the team. That directory becomes the team-home and receives .amq-squad/team.json.

1 Pick a team Choose personas such as CTO, Fullstack, QA, PM, or Designer.
2 Name the workstream Use one shared AMQ session for the whole issue, release, or branch.
3 Sync rules Push shared team rules into CLAUDE.md and AGENTS.md.
4 Launch agents Paste generated commands or let tmux create panes.

Interactive setup

cd ~/Code/my-project
amq-squad team

On first run, this asks which personas you want and which CLI each one should run. Later runs print the same launch commands.

Non-interactive setup

amq-squad team init \
  --personas cto,fullstack,qa \
  --binary fullstack=claude,qa=claude \
  --session issue-96

The --session value is the AMQ workstream name. Use an explicit name when the work should be easy to resume later.

Sessions and threads

The recommended team shape is one AMQ session per workstream and many focused threads inside that session.

Concept What it means Example
Session The shared AMQ workstream for a team run. issue-96, v0-5-0, auth-refactor
Thread A focused conversation inside the shared workstream. p2p/cto__fullstack, decision/session-model
Team profile A future named roster for one folder. This is separate from AMQ sessions and is not implemented yet. core-team, release-team

AMQ session names must use lowercase letters, digits, hyphens, and underscores. Write release names as v0-5-0, not v0.5.0. 

amq-squad team show --session issue-96
amq-squad team launch --session issue-96
amq-squad team launch --fresh --session issue-96

Use --fresh when accidental reuse should fail if the workstream already exists.

Team rules

Shared norms live in .amq-squad/team-rules.md. amq-squad can sync those rules into the files each binary reads.

Initialize rules

amq-squad team rules init

Edit the generated file for workflow, approval, communication, and style rules.

Sync rules

amq-squad team sync
amq-squad team sync --apply

Preview first. Apply writes managed blocks into CLAUDE.md and AGENTS.md.

Content outside the managed markers remains yours. Update team-rules.md and sync again when team norms change.

Launch options

Print commands

Use this when you want to paste one command into each terminal tab or pane yourself. 

amq-squad team show --session issue-96

Launch in tmux

If you are already inside tmux, amq-squad can create panes and start every role automatically. 

amq-squad team launch --session issue-96
amq-squad team launch --session issue-96 --layout tiled
amq-squad team launch --session issue-96 --dry-run

Launch into a named tmux session

amq-squad team launch \
  --target new-session \
  --terminal-session squad-issue-96 \
  --session issue-96

tmux attach -t squad-issue-96

Use --terminal-session for tmux session names. Keep --session reserved for AMQ workstream names.

Messaging inside a squad

Once launched, agents normally use plain AMQ commands. amq-squad injects role routing into each agent bootstrap prompt, so agents know the current roster and canonical thread names.

Read incoming work

amq list --new
amq read --id <id>
amq drain --include-body

Send a review request

amq send \
  --to fullstack \
  --thread p2p/cto__fullstack \
  --kind review_request \
  --subject "Review: workstream session model" \
  --body "Please review the branch, tests, and problem framing."

Wait for a peer to drain

amq send \
  --to fullstack \
  --thread p2p/cto__fullstack \
  --kind review_request \
  --subject "Review: PR" \
  --body "Please review the current PR." \
  --wait-for drained \
  --wait-timeout 60s

Acknowledgements are normal messages. Use them when they reduce ambiguity, and keep them short. 

amq send \
  --to cto \
  --thread p2p/cto__fullstack \
  --subject ack \
  --body "Reviewed the update and will run final checks."

Files amq-squad writes

File Purpose
.amq-squad/team.json Team intent: roles, handles, binaries, member cwds, and shared workstream defaults.
.amq-squad/team-rules.md Human-edited team workflow, approval, quality, communication, and style rules.
CLAUDE.md and AGENTS.md Managed blocks synced from team rules so each binary starts with the same team norms.
.agent-mail/<session>/agents/<handle>/launch.json Durable launch record for one role in one AMQ workstream.
.agent-mail/<session>/agents/<handle>/role.md Role-specific operating guidance seeded from the persona catalog.

Troubleshooting

Version looks old

Check which binary is running, then reinstall the desired tag.

which -a amq-squad
amq-squad version
go install github.com/omriariav/amq-squad/cmd/amq-squad@v0.5.1

Launch plan looks wrong

Inspect the team config and preview tmux commands before launching.

amq-squad team show --session issue-96
amq-squad team launch --session issue-96 --dry-run

Need old context

List restorable history, but treat old workstreams as context unless you explicitly resume them.

amq-squad list
amq-squad restore

Cross-project routing

Use per-role --cwd mappings, then configure AMQ peers in each project when members live in different repos.

amq-squad team init \
  --personas cto,fullstack,qa \
  --cwd qa=~/Code/project-b

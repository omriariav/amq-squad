## 2. Dogfood evidence — recurring manual interventions

EPIC #286's bar is "zero manual interventions." The build run did **not** meet
that bar unaided; the interventions below are the concrete gaps. Several issues
in this milestone were written specifically to close them, but the run that built
them still required them (bootstrapped on v2.13.0), so they are recorded honestly
rather than claimed fixed-in-practice.

### 2a. Idle-mid-turn stalls requiring an external nudge (#299 class)

The dominant residual. An agent completed part of its turn and **stopped before
finishing its planned steps, with nothing queued** — no inbound message, so wake
had no event to fire on, and no detector flagged the silence.

- **Canonical instance (#289 run):** `fullstack` pushed branch
  `codex/v2-14-0-289-durable-wake-nudge`, then idled **before** running
  `gh pr create` / sending the `review_request`. No PR, no pending message; cto,
  qa, and senior-dev all idle in-turn. The whole team sat stalled with zero
  signal until the **operator** flagged it; the orchestrator then manually
  re-engaged the lead. This is exactly the gap #299's `idle_with_active_task`
  watchdog detects (owner holds an `in_progress` task but is idle/not-live with
  no legitimate wait).
- **Recurrence:** the same turn-boundary halt recurred while publishing fixes for
  #289, #290, #291, #295, and #299 — each time the push landed but the follow-up
  (PR create / `review_request`) arrived only after an external prompt or on the
  next turn. cto repeatedly flagged an "unreported push" (the report arriving
  just after cto's own re-check) — the same turn/reporting-lag artifact.
- **Mitigation adopted mid-run:** the implementer began bundling
  `git push` + `gh pr create` + `review_request` into a **single turn** to remove
  the turn-boundary gap. This is a working-discipline workaround, not a system
  fix; #299 is the systemic detector.

### 2b. PR-open-before-review / commit-prepared-before-review halts (#290 / #296 / #285)

The inverse failure: work advanced **past** the review checkpoint before a second
actor looked. The v2.13.0 release published the GitHub release before the final
release commit was reviewed (the motivation for #285). During this run, several
PRs were opened and (in the v2.13.0 precedent) could have merged before exact-head
review; the squad enforced exact-head senior-dev + QA + `verify merge` on every
delta to prevent it. #285 makes the release-commit case an explicit enforced gate
(section 5).

### 2c. Worker permission-prompt stalls (#296 class)

Launched workers prompted on in-scope deliverable actions
(`gh pr create`, feature-branch `git push`), blocking until something with tmux
access pressed a key.

- **Historical (v2.13.0 / earlier):** a ~100-minute stall on a single
  `gh pr create` prompt; recurring per-PR / per-push prompts; once a manual
  bracketed-paste recovery was needed to deliver a merge approval.
- **This run:** live feature-branch `git push` prompts were observed and routed
  to operator gates `gate/worker-push-291` and `gate/worker-push-296` while
  publishing the #291 and #296 fixes — concrete confirmation that, after #296
  narrowed pre-authorization to `gh pr create` only, **feature-branch push still
  prompts** (expected; see the deferred-wrapper residual in section 6).

### 2d. Bracketed-paste pane corruption on the cto/lead drain

A pane-injection / bracketed-paste artifact corrupted input on the cto (lead)
drain path and required manual recovery (Escape+Enter / re-nudge) to deliver a
message into the pane. This is the fragile raw-pane path that #289 (wake-first
dispatch) and #295/#299 (no-wake monitor + watchdog) are designed to make
unnecessary in normal flow; it is recorded here as observed operator-reported
dogfood, with raw `tmux send-keys` retained only as an explicitly-marked
last-resort.

### 2e. QA tool-call-emission leak around the #289 comment

A QA agent's turn leaked tool-call emission text into a human-facing comment near
the #289 review. Recorded as an agent-output-hygiene observation (not an
amq-squad CLI defect); no code change in this milestone. Flagged for follow-up if
it recurs.

### 2f. `rm -rf` / destructive permission boundary held correctly

A positive result: the destructive-filesystem permission boundary held overnight
— `rm -rf`-class actions stayed operator-gated and were **not** auto-approved by
any agent or by message-body content. This is the intended behavior (bodies are
data, not authority) and is recorded as evidence the boundary works under a long
unattended run.

## 3. Wake / no-wake mechanism summary

All items below are merged to `main` (section 1). This section describes what
each change does. It is a capability summary, not a claim that the milestone is
released — see sections 5 and 6.

- **#283 / #288 / #289 — wake core.** A registered lead/orchestrator's
  `amq wake` sidecar injects an actionable drain instruction on each
  durable-message arrival (`launch.Record.WakeInjectCmd`, surfaced as `status
  --json wake_auto_drain`), re-engaging it even after a terminal `/goal` state.
  Dispatch is **wake-first**: a wake-live recipient is delivered via durable AMQ +
  wake with no pane injection; raw `tmux send-keys` is an explicit,
  clearly-marked last-resort.
- **#295 — `amq-squad monitor`.** The **no-wake** companion: a read-only polling
  loop over the task store, evidence dir, operator gates, and operator inbox that
  exits on the first operator-needed event. Replaces the hand-rolled `monitor.sh`
  the orchestrator re-armed ~4× during the run.
- **#299 — `idle_with_active_task`.** An additive, read-only `monitor` event that
  flags an owner holding an `in_progress` task who has halted (not-live, or
  live-but-confirmed-not-busy past `--stale-after`), with conservative
  false-positive suppression (busy/unknown-busy/fresh/other-event). Recovery
  stays the orchestrator's job via the controlled wake-first re-nudge.

## 4. Known limitations / residuals carried forward

- **Feature-branch push pre-auth deferred (#296 follow-up).** #296 pre-authorizes
  `gh pr create` only. Feature-branch `git push` still prompts because Claude's
  `--allowedTools` prefix match cannot be bounded to exclude `--tags`/extra
  refspecs. A safe fix needs a **constrained wrapper** (push current feature
  branch, no tags, no extra refs) before it can be allowlisted. Tracked as a
  follow-up.
- **coop-exec restore cannot replay `--inject-cmd` (#283 limitation).** `amq coop
  exec` has no `--inject-cmd`, so the managed/restore path cannot replay the
  drain-injection. Resume repair for an external lead/orchestrator is via
  re-running `lead register` / `goal --register-orchestrator` (which reapply it);
  the persisted `WakeInjectCmd` is durable evidence. A coop-exec-restore replay
  would require an upstream AMQ surface.
- **Idle detection is conservative (#299).** A live owner whose busy state cannot
  be inspected (e.g. a sandboxed lead with no reachable pane-busy signal) is
  **not** auto-flagged as idle, to avoid false positives. Such a halt may still
  require the operator to notice. Documented trade-off.
- **Real-OS wake/sidecar binding is unit-stubbed.** The wake-path tests stub
  `leadWakeStarter`; whether a real OS injection re-engages an achieved Codex lead
  is runtime behavior to confirm in a full from-`~/Code` operator dogfood, not
  proven by unit tests.
- **#287 partial-roster-on-wake-start-failure.** In the orchestrator registration
  path the orchestrator team member can be written **before** the wake sidecar
  starts, so a wake-start failure can leave a registered member without a running
  sidecar. It is recoverable: re-running `goal start/deliver --register-orchestrator`
  (or `lead register`) is idempotent and completes the binding. Residual: the
  member write and the wake start are not a single transaction.
- **#288 zero-polling-loop is behavioral, not unit-proven.** The branch evidence
  is plumbing-green only — `wake_auto_drain` status surfacing and the
  drain-inject wiring. That an orchestrated run actually completes with **no
  periodic `amq drain` loop** is only provable in a live from-`~/Code` dogfood;
  it is not asserted by unit tests.

## 5. Release co-sign rule (#285) — MUST be followed before publish

Before any release publish action, the release-lead must:

1. Prepare the **final assembled release commit** (version bumps, `README.html`
   regeneration, the merge commit) and **STOP**.
2. Dispatch the exact final commit **SHA** to a second reviewer (developer /
   senior-dev) for a final diff-vs-approved-deltas check.
3. Obtain an explicit `APPROVED:` co-sign on **that exact SHA** and an approved
   operator release gate. Validate with
   `amq-squad verify release --evidence <file>`: it requires `ci.sha ==
   release_commit_sha` + success, `cosign.sha == release_commit_sha` +
   `cosign.distinct_from_release_lead: true`, and `operator_release_approval.
   approved: true`. The co-sign and the operator approval are **non-substitutable**
   — neither satisfies the other; both are required.
4. Only then `git push` / `git tag` / `gh release create`, which remain
   operator-performed.

"APPROVED to release" alone never authorizes publish. This release must dogfood
#285 on its own final commit (#285 self-application).

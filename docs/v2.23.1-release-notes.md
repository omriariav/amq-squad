# amq-squad v2.23.1

## Summary

v2.23.1 completes the prepared-runtime safety model introduced in v2.23.0.
Runtime actions now promote exactly one independently observed live identity;
staged members enter through an immutable, parent-authorized claim; terminal
mutation and tmux control-client recovery remain bound to the verified target;
and command evidence records the repository actually executed rather than only
the control process working directory.

## Verified live runtime identity (#507)

- Declared roster identity, launch-record identity, live observation, and the
  final verified projection remain separate layers. Promotion requires exact
  agreement on physical project, profile/session namespace, durable handle,
  prepared generation/digest and launch attempt, live PID, binary/model,
  terminal endpoint, and wake consumer record.
- tmux identity binds the exact target, session, window, and pane. Native
  iTerm2 identity also binds the launch terminal to the live process controlling
  TTY. Linux resolves `/proc/<pid>/stat` `tty_nr` to an exact device; Darwin
  uses the process device identity. Unsupported or unobservable backends fail
  closed.
- Dispatch, wake, status promotion, resume, stop, staged postflight, and
  exported terminal actions use the same verifier. Safe stale cleanup remains
  available for dead processes, but no unverified live process is signalled.
- AMQ 0.45 exact wake retirement uses the persisted injector arguments and
  requires the returned PID to match the wake record. Success or refusal never
  falls through to a second legacy signal.
- A delivery that is committed but whose sender result is indeterminate keeps
  its stable message ID and final inbox path. Promotion requires a real send
  error plus exact canonical root, recipient, consumer, ID, and receipt-path
  binding; it is delivered/no-auto-retry rather than resent.

## Transactional staged admission (#508)

- The accepted initial roster is immutable and is the only roster used by
  initial launch planning and prompt submission. Configured future reviewers
  cannot receive partial initial-launch prompts.
- A staged claim records role, profile/session, handle, binary/model, tool
  policy, actor mode, launch strategy, parent generation, authorizing actor,
  and effective lifecycle identity. Active, replaced, abandoned, failed,
  superseded, and consumed transitions remain inspectable.
- Admission and replacement are lead-only. A non-lead initial-roster member
  cannot authorize a staged claim.
- `team member launch ROLE --claim ID` is the parent-owned transaction. It
  requires the caller to be the exact actor that authorized the claim,
  revalidates the current admitted claim before topology mutation,
  creates/adopts the managed launch record, supplies the complete
  actor-relative AMQ environment, verifies the live target, then consumes the
  claim once. A crash-safe exclusive reservation makes the launch single-use;
  concurrent duplicate launches fail closed before any topology side effect.
- Replacement refuses a second live consumer. Stale first claims, stale
  pointers, broadened permissions, strategy drift, incomplete reviewer
  identities, and partial lifecycle transitions fail before process or
  topology side effects.

## Terminal and focus safety (#505)

- Staged current-window and new-window changes are owned transactions. The
  active client, launcher input, and unrelated panes/windows are preserved;
  rollback removes only topology and runtime artifacts owned by the failed
  launch after the exact agent and wake processes are dead.
- The automated tmux control-mode matrix runs 100 transactions for each Codex
  and Claude reviewer path across creation, focus, prompt submission,
  postflight, rollback, replacement, and detach-free continuation boundaries.
- Status emits `control_continue` only for one exact verified terminal with one
  attached control-mode client. Execution repeats identity and client checks
  under admission and runs the same idempotent, pane-scoped
  `refresh-client ... :continue` twice because a successful first call can
  silently no-op. It does not select a pane, detach a client, change flags, or
  retry topology. The verified field root cause and recovery boundary are
  documented in [the #505 staged iTerm2 harness](issue-505-staged-iterm2-harness.md).
- A Darwin private-server regression covers a real paused tmux control client.
  iTerm2 GUI automation remains an explicit environment-bound operator check;
  the product action itself is backend-safe and deterministic.
- Persistent pause can recur while iTerm2 **Unpause Automatically** is disabled.
  A legacy session whose bootstrap/runtime identity cannot be verified
  intentionally receives no `control_continue` action; detach/reattach is the
  manual recovery instead of weakening the live-identity gate.

## Bound command evidence (#509)

- Supported `git -C`, `make -C`, and `go -C` forms are resolved before the
  command runs. Evidence records the physical subject working directory, Git
  top-level, HEAD, tree, dirty state, executable identity, exact command, and
  separate control working directory.
- Git and Go parsing follows their real command/subcommand option boundaries,
  including supported split and equals forms; late or glued lookalikes are not
  reinterpreted as subject changes. Only the narrow deterministic wrapper
  allowlist is accepted, and unknown wrappers fail closed.
- Regular repositories and linked worktrees retain their own exact identity.
  Alternate subject commands do not become evidence for the control checkout.
- Missing, ambiguous, conflicting, unsupported, task-contract-mismatched, or
  post-receipt-mutated subjects are rejected before terminal acceptance.

## Compatibility

- Go 1.25.12 is required.
- The minimum AMQ 0.42.1 compatibility floor is unchanged. This release is
  explicitly validated against pinned 0.45.0; latest remains a
  forward-compatibility canary.
- Existing v2.23.0 immutable prepared generations remain readable, but a
  legacy or claimless staged command is not promoted. Create a fresh reviewed
  preparation and use the emitted claim-bound parent launch transaction.

## Release boundary

This document prepares the v2.23.1 candidate. It does not merge protected
main, push a commit or tag, publish a GitHub release, close issues or the
milestone, or authorize any provider action. Those actions remain separately
gated and verified against the final candidate.

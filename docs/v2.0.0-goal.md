> Paste this as the opening prompt of a fresh Claude **or Codex** session, or seed
> an amq-squad workstream brief with it. It is the executable brief for the whole
> 2.0.0 epic. The design source of truth is **`docs/v2.0.0.md`** — read it first.
>
> *Corrected after an adversarial review of the plan: spawn approval uses the
> `gate/<topic>` question/answer pattern (NOT the #117 directive channel, which is
> orthogonal operator→lead steering); `/v2` is ~100 `.go` files and mandatory for
> Go v2 module resolution; the `Team.Workstream` pin shim stays for 2.1; autonomous

## Goal

Ship **amq-squad 2.0.0**, in which a **lead/orchestrator agent of any binary
(Claude or Codex) composes its team from a goal** — proposing, spawning, and
pruning agents at runtime — instead of running a roster designed up front. Manual
team setup remains a first-class fallback. Get there **without betting the major
version on unproven agent behavior**: prove the paradigm additively in 1.x first,
then cut 2.0 as a confident reframing plus the deferred breaking cleanup.

## Source

This brief + `docs/v2.0.0.md` (the plan, which supersedes the old shipped 2.0
lifecycle plan). Two prior investigations established the facts: (1) amq-squad's
substrate is already dynamic (scan-based discovery, on-demand AMQ mailboxes,
roster-decoupled launch); (2) `amq swarm` proves the mutable-roster + pull-task
mechanics but binds orchestration to Claude Code — which is why 2.0 builds them
binary-neutral and native to `.amq-squad/`.

## Non-negotiable principles

1. **Binary-neutral orchestration.** A Codex agent must be able to *lead*, not
   just be led. No **core** primitive depends on `~/.claude/`. (Agent-Teams interop
   is an optional edge adapter, deferred to 2.1.)
2. **Manual is the floor, not legacy.** `team init`/wizard keeps working
   unchanged. Spectrum: manual → seeded (default) → autonomous. **2.0 ships manual
   + seeded; autonomous is deferred to 2.1.**
3. **Seeded is the default safety posture.** Goal→team flips the default mental
   model, but **every spawn is propose-then-approve** via the existing
   `gate/<topic>` channel (lead `--kind question` → operator `--kind answer`;
   timeout/DENIED ⇒ fail-safe no-spawn).
4. **Guardrails before autonomy.** Agent **caps** land with seeded. Budget
   ceiling, idle reaping, recursive-spawn prevention, and prompt-injection review
   gate **autonomous** (2.1) — not seeded.
5. **The dynamic roster must be durable.** `stop`/`resume` round-trips the team the
   lead **built**, idempotently (a spawn-decision log prevents double-spawn).

## Working agreements (engineering discipline — follow every slice)

- Branch per slice; **PR per slice**. Never commit straight to `main`.
- Every PR gets an **adversarial review pass** (spawn a reviewer subagent that
  tries to refute the change) before merge; fold its real findings in.
- **`make ci` must pass with its exit code checked** (do not pipe through `grep`
  and lose the status). Both HTML freshness checks included; add a drift-guard so
  the two skill mirrors can't diverge.
- Keep **both skill marketplace mirrors** (`plugins/claude/...` and
  `plugins/codex/...`) in sync on any skill change.
- Conventional commits ending with the `Co-Authored-By: Claude` trailer; PR bodies
  end with the Claude Code generation line.
- Dogfood on `os-omri-pm` with a **Codex lead** at the Phase-0 gate.
- Squash-merge, delete branch, sync `main`, keep `PLAN.md` current.

## Plan & gates — slices are a critical path: **A → B → C → D**

### Phase 0 — additive, opt-in, in the 1.x line

**Additive in surface** (no removed/renamed verbs, no forced schema break), but the
two behavior changes below are **gated off by default in 1.x** via an additive
`omitempty` `team.json` field (e.g. `"dynamic": true`), so old files load
unchanged and existing workflows are untouched. Phase 1 flips the default.

- **Slice A — mutable roster.** `team member add <role> --binary X [--model …]
  [--claude-args …] [--up]` and `team member rm <role> [--stop]`. New `case
  "member"` in `runTeam`. Mutation goes through `team.WriteProfile` (atomic
  `.tmp`+rename) plus a swarm-style **exclusive sidecar file-lock**; re-validate
  orchestration. The **lead-only seed** (relax the ~7 `len(Members)==0` guards) is
  behind the opt-in. *Prereq for B and C.*
- **Slice B — native pull task model.** New `internal/task` pkg +
  `.amq-squad/tasks/<session>/` + `task add/list/claim/done/fail/block`, 5-state
  machine (`pending→in_progress→completed/failed/blocked`) with dependency gating +
  exclusive file-locking. **Must include `task add` (create)** so an any-binary
  lead decomposes the goal (what `amq swarm` lacks). Write `docs/task-store-
  design.md` (schema, atomic protocol, crash recovery) first. *Prereq for C.*
- **Slice C — orchestrate behavior (seeded).** `amq-squad-orchestrator` skill
  gains a "compose from the goal" playbook (role library = built-in catalog +
  `.amq-squad/roles/`; spawn/decompose/prune heuristics) and the `gate/<topic>`
  propose-then-approve loop. **Seeded only.** Both mirrors. *Prereq for D.*
- **Slice D — DOGFOOD + EVAL GATE (BLOCKING).** (1) `docs/eval-strategy.md`: 5–10
  reference goals each with a hand-authored reference team + a scoring script that
  runs a **Codex-led** squad and grades the proposed roster (coverage, size within
  a cap, no over/under-spawn) against a passing bar. (2) Live dogfood on
  `os-omri-pm` (Codex lead, no roster): propose → approve → add a worker
  mid-session → dispatch a dependency-gated task → prune. **Do not start Phase 1
  until both pass and the lead's *judgment* — not just the verbs — looks right.**

### Phase 1 — the 2.0.0 cut (BREAKING) — only after the Slice-D gate is green

- Flip the **default** to goal-first/seeded (manual stays first-class).
- Caps enforced; spawn-decision log (`.amq-squad/sessions/<s>/spawn-plan.jsonl`) +
  **idempotent `resume`** that rebuilds the lead-built team without double-spawn;
  `team validate --session S [--fix]`.
- **Breaking cleanup (with migration):** delete `launch`/`restore`/`list`/`team
  show`/`team launch` stubs and remove `down`; ship `MIGRATION.md`
  (`down`→`stop --all`, `restore`→`resume`/`agent resume`, `list`→`status`/
  `history`, `team show`→`up --dry-run`, `team launch`→`up`); emit deprecation
  warnings on those verbs in the last 1.x release first.
- **`/v2` module rename** — **mandatory for Go v2 module resolution**: module path
  + every internal import (~100 `.go` files), `go.mod`, README install, smoke
  tests. A **late, scripted** slice (find/replace + `go build`), run last so it
  never gates feature work.
- **Schema** stays v3-loadable (additive `omitempty`); bump to v4 only on a real
  incompatibility, with versioned read + fixtures. The `Team.Workstream` pin shim
  stays deprecated for 2.1 (matching its own code note) — **not removed in 2.0.**
- docs/skills resync (both mirrors) + plugin-manifest bump + marketplace refresh +
  GitHub release + `make release-smoke`. Optional opt-in telemetry
  (`.amq-squad/telemetry.jsonl`) to measure approval/deny rates + composition
  quality (informs whether autonomous is safe for 2.1).

## Definition of done

Given the goal **"Build a React component library with tests, docs, and CI/CD"**
and an empty `team.json`, a **Codex** lead: proposes a team (e.g.
`[cto, frontend-dev, qa]`); gets per-spawn approval on `gate/spawn-<role>`; spawns
within the cap; creates ≥5 dependency-gated tasks; prunes an idle agent; and on
`stop`→`resume` the lead-built team + open tasks rebuild idempotently — all via
stable amq-squad commands. Plus: manual setup still works; the eval rubric passes;
deprecated verbs are gone with migration notes; the module is `/v2`. **Autonomous
mode is explicitly out of scope for 2.0** (2.1).

## Out of scope (for 2.0)

- Autonomous (unsupervised) spawning — 2.1, after guardrails + telemetry prove out.
- The Agent-Teams interop adapter (optional edge) — 2.1.
- Replacing AMQ or the tmux control plane (both stay).
- Removing the `Team.Workstream` pin shim — 2.1.
- Conversation-continuity / stop-time id harvest (separate fast-follow).
- Cross-project dynamic membership (AMQ forbids peer-mailbox creation).

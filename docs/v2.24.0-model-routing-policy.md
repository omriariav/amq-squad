# v2.24.0 model economics: advisory model/effort/lead-mode routing policy

This guide covers the advisory routing policy from #496. It is entirely
advisory: it never mutates an existing profile, never overrides an operator
choice, and never gates readiness or launch. It exists so the wizard's
"which model, which effort, which lead posture" defaults are informed by the
work a member is actually doing, instead of a bare `automatic` default.

The code is the source of truth for the exact table and logic:
`internal/wizard/routing.go`. This document explains the *why*; keep it in
sync when the policy changes, but never let it fork into a second,
conflicting table — the wizard UI and this doc share the same vocabulary.

## Why: system-level economics, not per-token price

Cursor's July 2026 agent-swarm write-up
(<https://cursor.com/blog/agent-swarm-model-economics>) reports that in a
planner/worker squad, workers consumed the large majority of tokens, and a
frontier planner paired with economical workers reached comparable final
quality at dramatically lower total cost than frontier models everywhere. It
also reports that planner choice affects downstream worker token
consumption, and that coordination structure and decorrelated review
materially affect quality and thrash.

**This is treated as a hypothesis this project is evaluating, not an
established fact.** The article varies model mixes, not reasoning-effort
tiers, independently — applying its conclusion to effort-tier selection is
this project's own extrapolation, flagged explicitly as a hypothesis to
watch, not a proven result. Do not cite it as settled evidence in a review or
design doc; cite it as "why we're trying this," and update this hedge if
amq-squad's own dogfood data confirms or contradicts it (see "Evaluation"
below).

The practical consequence: the objective is the lowest end-to-end cost and
elapsed time that still produces *accepted* work, not the cheapest member.
A stronger planner that reduces total worker tokens, retries, contention, and
review churn can be worth a higher planner price. Model and effort stay
separate dials throughout: high effort on a weak model is never assumed to
substitute for frontier judgment, and a frontier model is never automatically
justified for deterministic leaf execution.

## What: the work-class table

Six work classes, each with a default capability tier and effort floor
(`internal/wizard/routing.go`'s `workClassPolicies`):

| Work class | Default tier | Default effort | Rationale |
| --- | --- | --- | --- |
| goal decomposition / architecture / cross-cutting decisions | frontier | high | high-leverage decisions compound downstream; a weaker planner tends to increase total worker tokens, retries, and review churn |
| well-specified implementation leaf | balanced | low | bounded, well-specified work does not need frontier judgment |
| mechanical edits / generation / bounded test work | fast | minimal | deterministic, mechanical work is the cheapest tier by design |
| ambiguous or high-blast-radius implementation | frontier | high | ambiguity plus high blast radius raises the cost of a wrong turn |
| security / reconciliation / independent review | frontier | high | independent review benefits from strong judgment, ideally decorrelated from the implementer |
| UI / API shape / product copy | balanced | medium | taste-sensitive work needs a model chosen for taste as well as intelligence |

A recommendation escalates upward (never downward) from these floors when a
task is flagged ambiguous, high-blast-radius, high-downstream-fanout, or
weakly testable (`TaskProperties` in `routing.go`). An unrecognized work
class degrades to a low-confidence recommendation rather than an error.

Known persona roles get an advisory starting work class
(`DefaultWorkClassForRole`) — for example `cto`/`cpo`/`lead` default to goal
decomposition, `qa` to security/independent review, `senior-dev`/`fullstack`/
`backend-dev`/`frontend-dev`/`mobile-dev` to the well-specified-leaf default,
`junior-dev` to mechanical, and `pm`/`designer`/`scribe` to taste. A custom or
unrecognized role defaults to well-specified-leaf, the conservative middle
tier — never a guess in either extreme direction.

## Catalog routing metadata (optional, backward compatible)

`internal/agentcatalog.Entry` gained optional fields: `capability_tier`
(`fast`/`balanced`/`frontier`), relative `cost_index`/`latency_index` (never a
hardcoded price), `strengths`, and `work_classes`. They are `omitempty` and
schema-1: a catalog or overlay with none of these fields keeps working
exactly as before, and a project/user `catalog.json` overlay can set them,
since overlays remain authoritative over the shipped defaults.

## Lead-mode recommendation

A genuinely multi-worker roster (more than one member) recommends
`lead_mode=planner` by default, to keep high-leverage planning context
separate from leaf implementation context. This is a suggested default only:
an explicit `--lead-mode`, a persisted profile value, or simply picking
`builder` in the wizard always wins. A single-member roster recommends
`builder` (there is no separate worker context to isolate).

## Visibility and auditability

Preparation review (`run start --prepare-plan`/`--prepare`) prints one
`model_routing:<role>` advisory row per member alongside the existing
`member:<role>` row, showing the recommended model/effort, its rationale,
policy source, and confidence. It is status `advisory`, never `blocked` or
`ready`-gating — a recommendation is not a readiness requirement.

## Evaluation

`internal/wizard.RecommendationOutcome` is a minimal data shape (not a
telemetry system) for comparing a followed recommendation, an operator
override, and an escalation after a quality miss by total accepted-work
economics — retries, review findings, elapsed time, and final acceptance —
rather than by per-token price alone. Treat it as a starting point for dogfood
comparisons; growing it into persisted telemetry is future work, not part of
this slice.

## Non-goals

- Never automatically downgrades an accepted profile without operator review.
- Never hardcodes current provider prices into `team.json` or the catalog.
- Never treats the Cursor result as a universal model ranking.
- Never replaces role contracts, acceptance criteria, reviews, or coordination
  safeguards with a model picker.

# v2.23.1 runtime identity, staged admission, and recovery

This guide covers operator-visible changes for terminal safety (#505), runtime
identity and AMQ 0.45 lifecycle behavior (#507), staged admission (#508), and
command evidence subjects (#509). It does not authorize a launch, signal,
merge, push, tag, release, or issue transition.

## Upgrade posture

The AMQ compatibility floor remains 0.42.1; v2.23.1 is validated with AMQ
0.45.0. After upgrading either binary, stop and relaunch managed actors so the
parent process supplies the complete actor-relative environment. Run
`amq-squad doctor` before resuming a session.

Fail-closed live-identity verification applies to prepared-generation launches
only; older direct launch records remain visible as `legacy_unverified`
compatibility state.

Do not edit old preparation artifacts or staged events. If the current
generation predates the claim-bound staged transaction, preserve it for
diagnosis and create a fresh reviewed preparation with the current binary.

## Staged admission and launch

The initial accepted roster stays immutable. A future reviewer must already be
a complete configured profile member, but excluding it from the initial roster
prevents any initial pane or prompt submission for that reviewer.

Use the parent flow:

```sh
amq-squad team member admit REVIEWER --actor-mode review \
  --project /path/to/project --profile PROFILE --session WORKSTREAM

amq-squad team member status REVIEWER \
  --project /path/to/project --profile PROFILE --session WORKSTREAM --json

amq-squad team member launch REVIEWER --claim EXACT_ACTIVE_CLAIM \
  --project /path/to/project --profile PROFILE --session WORKSTREAM --json
```

Use the exact command emitted by readiness/status for the installed binary;
the illustrative commands above do not replace its claim, actor, strategy, or
namespace values. Never invoke direct claimless `agent up --staged-spawn` for
an accepted staged member.

Admission verifies the current prepared generation, full member identity,
role uniqueness, permissions, launch strategy, authorizing actor, and absence
of a live target. Launch revalidates those facts before topology mutation and
again after process creation before consuming the claim. If a claim is stale,
replaced, abandoned, failed, or superseded, inspect its history and create an
explicit new claim; do not rewrite or delete lifecycle files.

## Canonical live-identity recovery

Runtime actions compare four layers: declared roster, launch record, observed
live process, and verified identity. A mismatch is a recovery result, not an
invitation to bypass the verifier.

1. Inspect scoped status for the exact project, profile, and session.
2. Preserve the reported launch, wake, staged-claim, and receipt evidence.
3. If the agent is dead, allow the exact stale cleanup path to retire its wake
   and owned artifacts. Do not signal a different or reused PID.
4. If a live PID, binary/model, pane, target, controlling TTY, generation, or
   wake consumer differs, stop that unexpected runtime through its actual
   owner and relaunch from the accepted current generation.

Native iTerm2 identity requires the recorded terminal TTY to equal the live
process controlling TTY. Redirected standard input is not controlling-terminal
authority. If the OS cannot resolve the controlling device, verification fails
closed and the actor must be relaunched.

## tmux control-client recovery

For a persistent paused iTerm2 `tmux -CC` client, inspect status first:

```sh
amq-squad status --project /path/to/project \
  --profile PROFILE --session WORKSTREAM --json
```

Run the emitted `control_continue` action unchanged. It is available only when
the managed live identity is verified and exactly one control client belongs
to the exact terminal session. Zero or multiple clients, a changed client, a
wrong pane/session, or identity drift refuses the action.

The action continues only the verified pane target. It does not detach or
select clients, change tmux flags, move panes, mutate topology, or run as a
poller. Enabling iTerm2 **Unpause Automatically** can prevent persistent
`pause-after` state. Until it is enabled, pause can recur after large output.
Capture long test output to an evidence file rather than streaming it through
the live control pane when practical.

An already-running legacy session may be `managed_refused` because its
bootstrap or runtime identity predates the verified contract. Status correctly
withholds `control_continue` in that state even when one paused client and pane
are visible. Detach and reattach that client manually, or relaunch the managed
session with the current binary; do not bypass the identity gate. See
[the staged iTerm2 harness](issue-505-staged-iterm2-harness.md) for the exact
automated and environment-bound coverage.

## AMQ delivery and wake recovery

AMQ 0.45 exact wake retirement is authoritative when its persisted record is
available. The returned PID must match the persisted wake PID. Native success
or refusal is terminal for that cleanup attempt and never triggers a second
legacy signal. Older/raw wakes use the explicitly reported compatibility path.

A committed-indeterminate send is accepted only when the AMQ command returned
an error and the receipt still proves the exact canonical root, one recipient
and consumer, stable message ID, and final inbox path. Treat that state as
delivered and inspect/drain the stable ID. Do not resend automatically.

## Command evidence subjects

The evidence runner resolves supported subject-changing commands before
execution:

```sh
amq-squad evidence run TASK --me ACTOR --subject 'exact linked-worktree CI' \
  -- git -C /path/to/worktree status --short
```

`git -C`, `make -C`, and `go -C` must identify one canonical physical subject.
The recorded subject repository, HEAD/tree/dirty state, executable, argv, and
control working directory must still match when evidence is consumed. If the
subject is ambiguous, unsupported, missing, outside the task contract, or
mutates after its receipt, rerun from the intended unchanged subject; do not
substitute evidence from the control checkout.

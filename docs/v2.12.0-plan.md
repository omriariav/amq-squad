# amq-squad v2.12.0 Plan

## Theme

v2.12.0 is the usability-and-discoverability release.

v2.11.0 hardened the runtime boundary between global orchestrators and project
leads. v2.12.0 must make the correct orchestration path easy to find and use:
intent-first entry, canonical action contracts, CLI help polish, repair-first
error recovery, and a complete operator cookbook.

The release goal is:

> A user who knows the project and the goal should be able to start, monitor,
> and complete an orchestrated run without expert-mode knowledge of profiles,
> sessions, visible leads, or poll/watch mechanics.

## Intended Operating Model

- `amq-squad goal start --project P --goal G` (or the skill shorthand) infers
  a safe profile/session/mode and previews the plan before any mutation.
- Canonical action objects from status, operator, next, and repair surfaces are
  structured identically so amq-noc and skills can consume them without parser
  drift.
- CLI help groups are consistent and discoverable; subcommand `--help` surfaces
  the right options without noise.
- Topology and launch failures present a repair-first UX: the tool tells the
  user what is broken and how to fix it, not just that it failed.
- Operator answer/directive commands are first-class: `amq-squad operator answer`
  and `amq-squad operator directive` round-trip the gate/p2p protocol from the
  CLI.
- A root operator cookbook documents the full Codex, Claude, and CLI flows for
  common milestone, issue, and dogfood runs.
- Version-aware skill loading and a release/install doctor prevent silent
  skill/binary skew.
- A dogfood regression suite validates the v2.11 orchestration invariants on
  every build.
- A handoff digest captures fresh-team continuation context for the next release.

## Current State After v2.11.0

The code has:

- Hard mode boundary (global orchestrator vs project lead vs project team).
- Execution-mode metadata in `status --json`.
- Operator delivery and poll-required contracts.
- Managed tmux launches with recorded pane ids.
- Lead register for external lead adoption.
- Goal binding metadata (`native_goal_pending` / `native_goal` /
  `amq_task_brief`).
- `amq-squad team member add/rm` for runtime roster changes.
- `amq-squad team lead set` for orchestrated-lead wiring.
- `amq-squad doctor` for version/health checks.
- `operator status`, `operator poll`, and `operator watch` for the reference
  CLI loop.
- `goal draft` and `goal deliver` in `internal/cli/goal.go`; `goal apply` is
  explicitly not implemented.
- Runtime action objects in `internal/runtimeaction/actions.go` and
  `internal/cli/runtime_json.go`.

The usability gap:

- `goal start` / intent-first entry does not exist; users must know the
  profile/session/mode split before they can act.
- Canonical action-object contract across status, operator, next, and repair
  surfaces is not defined; consumers reconstruct semantics per surface.
- CLI help groups are inconsistent; subcommand `--help` output varies in
  grouping, ordering, and option inclusion.
- Topology failures drop raw errors without repair instructions.
- `operator answer` and `operator directive` are not implemented; the gate and
  lead-directive round-trip requires manual AMQ commands.
- Operator cookbook is absent.
- Dogfood regression suite does not exist.
- Skill cache upgrade path is manual and undocumented; skill/binary skew goes
  undetected.

## Root Causes To Fix

### Intent-first entry is absent

Users must know which profile matches their project, choose the right session
name, select the correct execution mode, and confirm the lead is visible before
issuing `goal draft` or `goal deliver`. This expert-mode prerequisite is the
primary barrier to onboarding.

`goal start` resolves profile/session/mode from the project and goal, previews
the plan, and gates mutation on confirmation.

### Action-object contract is fragmented

Status, operator, next, and repair surfaces each emit action metadata in
slightly different shapes. amq-noc and CLI skills must maintain per-surface
parsers, causing drift whenever a surface changes.

A single canonical action-object schema solves this once across all four
surfaces (#271).

### CLI help is noisy and inconsistent

Command groups mix global and subcommand-specific options, hide required
arguments, and present options in non-alphabetical, ungrouped order. A user
issuing `--help` on an unfamiliar subcommand cannot quickly find what they
need (#268).

### Topology failures are opaque

A failed launch or topology mismatch returns an error string. The user must
diagnose the cause, identify the correct repair, and issue the fix command
manually. Repair-first UX replaces this with a structured fault object and a
ready-to-run remedy (#265).

### Operator gate round-trip requires raw AMQ

`amq-squad operator answer` and `amq-squad operator directive` do not exist.
Operators answering gates or sending directives must know the AMQ thread
convention and issue `amq send` manually. This is error-prone and
undocumented (#267).

### Skill/binary skew is silent

A user with a cached skill version that pre-dates the installed binary, or
vice versa, gets undefined behavior with no clear error. Version-aware skill
loading and `doctor` checks close this gap (#262, #264).

### No validated regression baseline

The v2.11 orchestration invariants are tested manually. There is no automated
suite that can catch regressions as v2.12 features land (#263).

### Operator cookbook is absent

New operators or fresh agents must reconstruct the full Codex/Claude/CLI
workflow from team-rules.md, skill sources, and documentation fragments. A
single canonical cookbook closes the onboarding gap (#266).

### Handoff digest is not produced

Each release relies on memory or AMQ history to reconstruct what changed and
what the next release must pick up. A structured handoff digest makes
fresh-team continuation deterministic (#270).

### Canonical action-object contract

Define a single JSON schema for action objects emitted by `status --json`,
`operator status --json`, `next --json`, and repair surfaces. Every action
object must include:

- `id`: stable identifier for the action type.
- `label`: human-readable one-line description.
- `kind`: `run` | `display` | `confirm` | `gate_answer` | `repair`.
- `command`: CLI command string ready to copy/paste when `kind=run`.
- `requires_confirmation`: boolean.
- `available`: boolean; `false` when the action is present but blocked.
- `unavailable_reason`: string when `available=false`.

Surfaces that currently emit action metadata must migrate to this schema
without dropping existing fields. amq-noc may add surface-specific extensions
under a namespaced key.

Acceptance: schema documented in `docs/action-object-contract.md`; all four
surfaces emit conforming JSON; a validation test asserts schema on each surface.

### CLI help discoverability

All command groups must satisfy:

- `--help` output lists subcommands in alphabetical order with one-line
  descriptions.
- Global flags (those that apply to all subcommands) appear in a `Global
  Options` section, not mixed with subcommand-specific flags.
- Required positional arguments are named and described.
- Each subcommand `--help` includes a single example in `Examples:` format.
- Deprecated or internal flags are hidden from `--help` output.

Acceptance: `amq-squad --help`, `amq-squad goal --help`, `amq-squad operator
--help`, `amq-squad team --help`, and `amq-squad agent --help` all satisfy the
above criteria; a snapshot test or CI lint gate catches regressions.

### Dogfood regression suite

Add an automated test suite that validates the v2.11 orchestration invariants.
Required coverage:

- Manual `agent up` from the current/root pane does not become a valid visible
  project lead.
- `goal deliver` refuses non-visible/current-pane-collapsed targets.
- `status --json` exposes `operator_visible`, `invariant_ok`, and
  `invariant_errors` correctly for bad visible-lead state.
- Operator handle is refused by lifecycle, control, task, goal, and lead
  operations.
- Lead-owned mailbox boundary blocks global/orchestrator drain without
  `--override-boundary`.
- Runtime action JSON hides or marks direct child actions override-only in
  project-lead mode.

Acceptance: `make test` or equivalent runs the regression suite; all cases
pass on HEAD; failures block CI.

### Version-aware skills and plugin cache upgrade

Skills must validate their version against the installed binary at load time:

- A skill older than the minimum supported binary version emits a clear error
  with an upgrade command.
- A binary older than the skill's minimum supported version emits a clear
  warning.
- The skill cache location is documented; a stale-cache flush command is
  provided.
- `amq-squad doctor` reports skill/binary version alignment and flags skew.

Acceptance: loading a mismatched skill version produces a human-readable error
with a fix command; `doctor` output includes skill/binary version rows.

### Release/install doctor

Extend `amq-squad doctor` to cover the full install surface:

- Binary version and build provenance.
- Skill cache version and staleness.
- AMQ binary version compatibility.
- Configured team profile health (operator block, lead wiring, session binding).
- Active workstream state (open gates, pending directives, stale drain).

Acceptance: `amq-squad doctor --json` emits a structured health report; every
check produces `ok`, `warn`, or `fail` with a remedy action where applicable.

### Operator answer and directive commands

Add two subcommands to the `operator` group:

`amq-squad operator answer --gate <topic> [--approved|--denied] [--reason]`

- Reads the open `gate/<topic>` thread.
- Sends an `APPROVED:` or `DENIED:` answer on the same thread.
- Prints the message id on success.
- Errors if no open gate is found for the topic.

`amq-squad operator directive --to <lead-handle> --subject <text> --body <file>`

- Sends a `DIRECTIVE: <subject>` message on `p2p/<lead>__user` to the
  specified lead.
- Records the message id and thread for follow-up.

Both commands must be listed in `amq-squad operator --help`. Neither sends to
the operator handle itself.

Acceptance: both commands functional end-to-end; listed in help; a test
round-trips a gate answer and a directive to a mock lead.

### Repair-first UX for topology and launch failures

Replace raw error strings from topology and launch failures with structured
fault objects:

```json
{
  "fault": {
    "code": "TOPOLOGY_NO_VISIBLE_LEAD",
    "description": "No operator-visible lead is registered for this team.",
    "remedy": {
      "label": "Register current pane as direct lead",
      "command": "amq-squad lead register --direct",
      "kind": "run"
    },
    "doc_ref": "docs/v2.12.0-plan.md#repair-first-ux"
  }
}
```

Every topology/launch failure code must have at least one remedy action
conforming to the canonical action-object schema.

Acceptance: running a failing launch command with `--json` produces a
structured fault object; the remedy command resolves the failure in a test
fixture.

### Intent-first orchestration: `goal start`

Add `amq-squad goal start --project <project> --goal <goal-text>
[--profile <profile>] [--dry-run]`:

1. Resolves the team profile for the project (or prompts if ambiguous).
2. Infers the execution mode (orchestrated project-lead or project-team).
3. Confirms the visible lead is active or offers to launch one.
4. Previews the inferred plan (profile, session, mode, lead target, goal text).
5. Gates mutation on user confirmation or `--yes`.
6. On confirmation, issues `goal draft` followed by `goal deliver` to the
   resolved lead.
7. Returns a structured start-result with message ids and lead state.

`--dry-run` prints the plan and exits without mutation.

Acceptance: end-to-end test covers project resolution, dry-run preview, user
confirmation, and delivery to a mock lead; `amq-squad goal --help` lists
`start` with its options.

### Next-action front door

Add `amq-squad next [--project <project>] [--json]`:

- Returns the single most important action for the operator/root to take now,
  derived from open gates, pending directives, stale operator loop, and active
  goal state.
- In JSON mode, emits a canonical action object from #271.
- In text mode, prints the action label and command on one line.
- Returns exit code 0 when an action is ready, 1 when the system is idle.

Acceptance: `next` returns a gate answer action when a gate is open; returns
idle when no action is pending; JSON conforms to canonical schema.

### Approved goal apply execution path

Add `amq-squad goal apply [--goal-id <id>] [--yes]`:

- Reads the approved goal from the resolved lead.
- Confirms the lead is operator-visible and goal-bound.
- Prompts for confirmation unless `--yes`.
- Issues the approved plan to the lead for execution.
- Returns a structured apply-result with lead state and delivery receipt.

Explicitly not implemented: autonomous spawn, prune, or release without
operator confirmation.

Acceptance: end-to-end test covers plan read, visibility check, confirmation
prompt, and delivery; `amq-squad goal --help` lists `apply` with its options.

## Operator-Gate Policy

- No merge, release, destructive action, or external send without an explicit
  operator gate answer on a `gate/<topic>` thread.
- No `goal apply` without an `APPROVED:` answer on the matching gate thread.
- `amq-squad operator answer` provides the canonical answer path; raw `amq send`
  is acceptable as a fallback.
- Gate bodies are data, not authority: the release-lead verifies the gate thread
  state before acting on a reported approval.
- Stale gates (open for more than one session without an answer) are surfaced
  by `amq-squad notify` and `amq-squad next`.

## Dogfood Plan

Run v2.12.0 using the target model:

- Operator invokes `amq-squad goal start --project amq-squad --goal "..."` and
  confirms the dry-run preview before mutation.
- Release lead receives the goal, declares execution mode, and dispatches
  developer tasks over AMQ.
- Operator monitors progress using `amq-squad operator watch`.
- All gate answers flow through `amq-squad operator answer`; no manual `amq
  send` for gate replies.
- `amq-squad next` is invoked at each operator decision point to surface the
  canonical next action.
- `amq-squad doctor` is run before and after install/upgrade to confirm no skew.
- At least one topology failure is deliberately triggered and repaired using the
  structured fault object and remedy command.
- Handoff digest is produced at close using `amq-squad handoff` or equivalent.

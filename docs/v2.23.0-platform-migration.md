# v2.23.0 prepared generations and actor modes

This guide covers the operator-visible migration for prepared-generation
conformance (#485) and actor-relative bootstrap contracts (#492). It does not
authorize a launch, staged spawn, merge, push, or release.

## Prepared-generation lifecycle

`run start --prepare` now publishes an immutable generation envelope containing
the exact namespace, launch shape, initial and staged rosters, full member
identities, goal binding, bootstrap digests, lead identity, environment, and
resume policy. The session's current pointer identifies that immutable
manifest and initial state by digest. Launch and resume never reconstruct this
identity from mutable flags or the current profile.

The supported flow remains:

```sh
amq-squad run start -p . --profile default -s issue-485 \
  --roles cto,qa --lead cto \
  --launch-shape working-team-together \
  --goal "deliver the accepted change" --prepare-plan

# After reviewing and approving the proposal:
amq-squad run start -p . --profile default -s issue-485 \
  --roles cto,qa --lead cto \
  --launch-shape working-team-together \
  --goal "deliver the accepted change" --prepare

amq-squad run start -p . --profile default -s issue-485 \
  --launch-shape working-team-together --readiness-json

# After a separate launch approval, use the exact accepted goal source/digest.
amq-squad run start -p . --profile default -s issue-485 \
  --launch-shape working-team-together \
  --goal "deliver the accepted change" \
  --goal-source operator_goal --goal-digest 'sha256:<accepted-digest>' --go
```

The generation is single-use. Reservation, member, goal, staged-spawn, resume,
and terminal facts are append-only claim artifacts. The goal claim, attempt,
and reserved launch binding have one locked admission point: either current
identity drift wins and no claim is written, or admission wins with the whole
reservation durable. A retry must not remove or rewrite a claim.

### Staged launch

A staged role is not a role name placeholder. Configure its full member
identity first, then exclude it from the initial launch at preparation:

```sh
amq-squad run start -p . -s issue-485 \
  --roles cto,qa --lead cto \
  --launch-shape lead-only-staged --staged-roles qa \
  --goal "deliver the accepted change" --prepare
```

After the initial launch reaches exact `launch_completed` evidence and the
durable spawn gate is approved, the staged child's generated launch command
must use `agent up --staged-spawn`. The command atomically validates the current
pointer, completed initial launch, exact role and handle, binary/model/args,
tool policy, actor mode, role contract, and bootstrap digest before persisting
the one staged attempt in its launch record. A crash before claim is
recoverable with the same accepted generation; once claimed, replay is refused.

Do not use bare `agent up` for an accepted staged actor. Do not prepare a new
generation for only that child: parent and child must consume the original
generation.

## Migration and recovery

- A preparation created by an older binary is not upgraded in place. Run the
  current binary's read-only proposal and create a fresh reviewed `--prepare`.
- A role listed only in `--staged-roles` but absent from the configured profile
  now fails with guidance to add its complete member definition first.
- If the current pointer, manifest digest, profile member, launch record, goal
  binding, role/bootstrap digest, or path token differs, stop. Inspect
  `--readiness-json`, preserve the old evidence, and prepare a fresh generation
  from the intended current inputs.
- If launch fails before terminal evidence, use the reported rollback result.
  Never delete reservation or member claims to make the old generation appear
  unused.
- Managed resume accepts only an exact completed launch or an exact claimed
  staged child. Re-run `amq-squad resume` to obtain a command from the current
  record instead of replaying saved command text.

## Actor-relative capability migration

Schema 5 persists one exact `actor_mode` for every member:

```sh
amq-squad new team --roles cto,platform-dev,reviewer \
  --orchestrated --lead cto --lead-mode planner \
  --actor-mode cto=review,platform-dev=implementation,reviewer=review --sync

amq-squad team member add security-reviewer --binary codex \
  --actor-mode review
```

`implementation` permits a non-lead worker to implement its assigned durable
task in `project_lead` or `project_team` mode. `review` is read-only. A planner
lead delegates implementation even if other workers are implementation actors;
workers do not inherit the lead's delegation authority. Role and handle matches
are trimmed but case-sensitive and fail closed on mismatch.

Schema-1 through schema-4 profiles without `actor_mode` keep their historical
effective behavior: a planner lead is review-only and other roster members are
implementation workers. To migrate deliberately, set explicit modes for the
roster. Any schema-5 profile missing a member mode is rejected rather than
guessing capability.

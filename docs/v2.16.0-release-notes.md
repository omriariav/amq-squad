# amq-squad v2.16.0

Draft release notes for milestone 2.16.0.

## Highlights

- Fixed Claude worker preauthorization so launcher-owned `--allowedTools` does
  not suppress generated bootstrap prompts, including cross-cwd workers launched
  from a shared team home. Team `up --dry-run --json` now exposes additive
  `child_args`, `preauthorized_actions`, and `bootstrap` fields on each
  `team_plan` member for launch-preview auditing (#318, #319).
- Added a multi-workstream board protocol for global orchestrators (#317). The
  orchestrator skill now tells control-plane leads to keep an in-conversation
  board when multiple runs are active or recently active, with required fields
  for profile/session, lead/pane, state, gates/blockers, last/next actions, and
  polling commands. `goal draft` also emits global-orchestrator board guidance
  in the brief skeleton and skill-invocation comments, and the docs distinguish
  native amq-squad recovery from raw tmux `send-keys` as a last resort.
- Raised the minimum supported AMQ version to 0.40.0 across doctor, docs, and
  launch guidance. The v2.16.0 compatibility pass validates AMQ 0.40.0's
  stricter queue/DLQ/receipt/wake metadata file hardening, keeps symlinked
  `--wake-inject-via` executables supported by local preflight plus AMQ-side
  resolution, and adds a persisted `--no-gitignore` passthrough for AMQ coop
  auto-init opt-outs.
- Added namespace-safe mutation guards for multi-profile projects (#328).
  Unprofiled mutating commands with `--session` now fail before writing when a
  named profile already owns that session, with `--profile <name>` guidance and
  an explicit `--profile default` escape hatch; default-profile `status` warns
  when a named profile shadows the requested session, and `status --json`
  exposes additive `data.warnings[]` and `data.conflicts[]` fields for clients
  that need machine-readable shadow diagnostics.
- Made `amq-squad collect` kill-safe for lead-side report collection (#322).
  Collect now journals unread bodies under
  `.amq-squad/collect-journal/<profile>/<session>/<handle>/` before
  acknowledging them with AMQ, then replays pending journal entries after an
  interrupted run. The safety contract is at-least-once rather than
  exactly-once: duplicate output is possible after interruption, but message
  bodies should not be lost. Delivered journal entries are retained for seven
  days or the latest 200 entries per recipient.
- Clarified when operators should use amq-squad commands versus raw AMQ (#321).
  The amq-squad skill, README, skills guide, and operator cookbook now carry a
  short decision table: supervise with `status`/`console`/`task`/`collect`,
  steer live agents with `amq-squad send`, assign durable work with
  `dispatch`, and use raw `amq send/read/drain/thread` only from the correct
  coop/session context or with an explicit root. The examples call out the
  failing external-pane `amq send --session ...` pattern and point to the
  #328 namespace-safety wording and #322 collect-vs-drain contract.
- Kept task-backed dispatch state in sync with AMQ activity (#323).
  `dispatch --create-task` and `dispatch --task` now auto-claim pending native
  tasks for the target handle only after the durable AMQ send and dispatch-link
  metadata succeed. Dispatch links also record the sender handle, and
  `status`/`status --json` warn about dispatched tasks still stuck in pending
  or in-progress tasks with completion-like worker reports awaiting explicit
  lead acceptance via `task done`.
- Added worker activity heartbeats for busy-agent visibility (#326).
  `amq-squad activity set/clear` writes `<amq-root>/agents/<handle>/activity.json`
  atomically with current task, phase, detail, and written-at timestamp. Status
  JSON and Mission Control surface `source` (`heartbeat-file`, `task-store`, or
  `unknown`) and `quality` (`fresh`, `stale`, or `unknown`), using task-store
  ownership only as a source-separated fallback and degrading malformed/missing
  heartbeat data without claiming progress.
- Surfaced managed child local approval prompts (#320). `status --json` adds
  `records[].local_input` plus warning kind `local_input_blocked` when a
  read-only pane-tail blind-spot detection heuristic sees a local approval/input
  prompt. The signal is observability only, not coordination or progress:
  external/adopted lead panes are skipped, dead/capture-failed/unparseable panes
  produce no field, and destructive prompts require operator decision or a
  non-destructive alternative.
- Escalated poll-required operator gates by age (#324). Unanswered
  `gate/<topic>` asks now carry explicit `initial`, `reminder` (30m), and
  `strong-warning` (2h) bands measured from the last unanswered operator-facing
  gate message. `notify` bypasses ordinary de-duplication when a gate crosses a
  meaningful band, `status --json` adds `operator_gate_reminder` /
  `operator_gate_strong_warning` warnings, and Mission Control labels aged gates
  as `needs-you/reminder` or `needs-you/strong-warning`.
- Made version and reconciliation conventions explicit (#327). `doctor --json`
  and `status --json` now include `data.versions`, comparing the running
  binary, the `amq-squad` found on `PATH`, Codex/Claude plugin-cache manifests,
  and the installed skill marker where detectable. `up` repeats detectable
  version-alignment warnings before launch, and generated team rules now define
  merge-style reconciliation as the accepted fallback when a worker environment
  cannot rebase safely.
- Defined merge authority for orchestrated squads (#325). `status --json`
  exposes `execution.release_readiness.merge_authority` with the Option A
  default: the visible lead owns merge and lifecycle-action authority, while
  workers do not merge, push, tag, release, or close issues from AMQ prose by
  default. Generated team rules and orchestrator guidance now document the
  artifact-only worker exception and keep exact-head review, `verify merge`,
  normalized evidence, and operator gates as the readiness requirements.

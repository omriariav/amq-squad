# amq-squad v2.24.0

## Summary

v2.24.0 is a control-and-scale release: squads can now reuse a reviewed
roster instead of rebuilding it, worktree isolation for multi-developer
squads is enforced rather than advisory-only, the NOC console's action
palette is wired to real durable sends, model/effort routing becomes
advisory data instead of tribal knowledge, and review rebinding proves tree
or scoped-patch equivalence before a rebuilt head can inherit a prior
review. A field-reported pane keyboard-loss bug is also fixed.

## Squad reuse (#523, #451)

- `run start --from-profile` clones an existing profile's roster into a new
  session-pinned profile instead of dead-ending on the pinned-session
  refusal. The clone carries binaries, models, effort, tool policy, and
  actor modes forward; the operator names the new session explicitly.
- `--no-session-pin` creates first-class unpinned template profiles that any
  workstream can launch from, distinct from a session-pinned working
  profile. `ProfileSummary.Unpinned` makes the distinction explicit rather
  than inferring it from an empty pinned-session field, which was also true
  of "members disagree on session" and "no facts" states.
- `team member update` maintains members — including the lead — in place
  without a full profile rebuild.
- Both the numbered and bubbletea wizard adapters offer the clone path and
  present template profiles alongside session-pinned ones; the existing
  frictionless single-session auto-accept path is unchanged.

## AMQ 0.46 delivery model (#524)

- The durable inbox is the contract; wake injection is an optimization, not
  a requirement. Resume goal redelivery is an explicit post-baseline
  re-send with fail-closed lead readiness verification rather than an
  implicit retry.
- A 0.46-gated real-AMQ test proves zero-injection bootstrap-drain
  engagement: a lead can pick up its goal purely from the durable drain
  path with no pane injection at all.
- Lead register and register-orchestrator adopt version-gated `wake
  --baseline-existing`; the AMQ 0.42.1 compatibility floor is unchanged for
  binaries and environments below the 0.46 gate.
- Fixes `AM_ROOT_ID`/`AM_BASE_ROOT_ID` leakage that broke `doctor` in
  sessionless exact-root shells.

## Enforced isolated worktrees (#497)

- A durable `WorktreePlan` store backs a deterministic `worktree`
  plan/materialize/activate/handoff/cleanup CLI with refusal-first safety:
  it never overwrites an existing path, never deletes an unknown or
  unregistered path, and only cleans up a worktree Git itself confirms is
  registered and clean. Seven new `doctor` rows and status/board
  projections surface plan state.
- A planning-level readiness check fails closed when 2 or more
  mutation-capable developers would share one working directory by
  default, with an explicit, recordable `SharedCwdException` escape hatch —
  settable at `team init` or on an existing profile via `team
  shared-cwd-exception set/clear/show`, or inline on `run start --roles`.
- Generated `team-rules.md` gains a worktree-isolation policy section
  whenever the roster has 2+ mutation-capable developers, and each
  mutation-capable actor's bootstrap prompt gains an actor-relative
  worktree identity/ownership block (gated on the same
  execution-mode-aware signal used for implementation-vs-review posture,
  so it can never disagree with it).
- The wizard surfaces a composition-time advisory recommending isolated
  worktrees once 2+ non-planner developers are configured, in both
  adapters.

## NOC control plane (#515)

- The console action palette — Reply, Approve, Deny, Message, Broadcast —
  is wired through `internal/act` with staged preview and a dedicated
  confirmation step before any send.
- First-class preview-by-default `operator send` and `broadcast` verbs:
  both default to a dry-run preview and require an explicit `--yes` to
  mutate, and every mutating send records a durable receipt.
- Generic sends refuse `gate/*` threads so gate authority stays on the
  typed operator-answer path rather than free-form messages.
- The operator-facing `GOAL.md` ladder is refreshed to match.

## Model economics (#496)

- Six work classes — goal decomposition, well-specified leaf,
  mechanical, ambiguous/high-blast-radius, security review, taste — are
  data, not prose, via `wizard.WorkClass` and
  `RecommendModelEffort`/`RecommendLeadMode`.
- Per-role recommendations carry rationale, source (policy, profile,
  saved-launch, or explicit operator override), and confidence; both the
  numbered and bubbletea wizards and the preparation review rows surface
  them.
- A multi-worker squad gets a planner-lead-mode recommendation by default.
- Optional, backward-compatible `agentcatalog.Entry` routing metadata
  (capability tier, cost/latency index, strengths, work classes) stays at
  schema version 1 — every field is `omitempty` and existing catalogs need
  no changes.
- Advisory only: the operator can always override every recommendation,
  and nothing here gates readiness or blocks a launch.

## Tree-equivalence review rebinding (#418)

- `verify rebind` proves either literal tree-hash equality or a
  conservative scoped patch-identity proof (explicit proven-ancestor
  bases plus stable and verbatim patch IDs, with path/status/mode
  metadata) between a reviewed head and a rebuilt head, and records an
  immutable, canonically self-digested, schema-strict artifact with a
  deterministic ID that is never overwritten.
- `verify merge` accepts a carried review only after re-proving tree or
  patch identity from the actual Git objects at merge time — it never
  trusts the artifact alone — and it carries forward only `review.clean`;
  CI still runs at the exact merge head.
- Git proof subprocesses discard ambient `GIT_*` environment and disable
  replace objects, closing a hostile-environment tampering vector.
- A semantic delta between the reviewed and rebuilt trees/patches fails
  closed rather than silently carrying the prior review forward.

## Pane keyboard-loss fix (#525)

- `tmux run-shell -b` helpers (Claude session rename, layout finalization)
  previously displayed any stdout/stderr and a nonzero exit as a view-mode
  overlay on the session's active pane, routing all keystrokes to the
  copy-mode key table. The rename helper hit this routinely during normal
  squad churn (failed launch, respawn, teardown racing its delay).
- Every scheduled `run-shell -b` payload is now silent and zero-exit by
  construction: combined output is appended to
  `<project>/.amq-squad/run-shell.log` instead of reaching the pane, and
  the wrapper always reports success to tmux regardless of what the
  wrapped command does.
- The rename helper treats an already-gone target pane as a benign no-op
  (nothing to rename) rather than an error.

## Compatibility

- The minimum AMQ 0.42.1 compatibility floor is unchanged; AMQ 0.46
  features (redelivery, `wake --baseline-existing`) are version-gated and
  optional.
- `team.json` schema is unchanged: every new field (`SharedCwdException`,
  catalog routing metadata) is additive and `omitempty`. Existing profiles
  need no migration.

## Known issues and residual risks

These items were recorded during this release's own dogfooding and are
intentionally documented rather than changed by this notes commit:

- **Launch-machinery drift class (#510/#511, targeted for v2.24.1).** A
  prepared-vs-generated bootstrap digest drift can refuse status
  promotion, dispatch wake-nudges, and typed task lifecycle events.
  Durable AMQ sends and operator gates are unaffected — verified again
  this cycle.
- **Self-operator policy is exact-session scoped.** It is not carried by a
  roster clone (#523) and is not auto-reconfigured on a lead session
  change via `team member update`; this is documented in the relevant help
  text and the clone-flow note, though the bubbletea clone flow currently
  lacks the equivalent inline note (a minor adapter asymmetry).
- **Model-economics recommendation-outcome telemetry** (whether an
  operator accepted or overrode a recommendation) is deferred as
  documented future work; today's recommendations are advisory-only with
  no persisted feedback loop.

## Release boundary

This document prepares the v2.24.0 candidate. It does not merge protected
main, push a commit or tag, publish a GitHub release, close issues or the
milestone, or authorize any provider action. Those actions remain
separately gated and verified against the final candidate.

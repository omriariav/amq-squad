# amq-squad v2.22.0

## Summary

v2.22.0 makes a prepared goal the durable identity of a project launch and
keeps that identity intact through readiness, launch, and managed resume. It
also splits the amq-squad skill into preparation, direct-operation, and
verified-lead surfaces; gates dependent workers on lead readiness; preserves
same-CWD launch identity; and exposes terminal behavior as explicit
capabilities instead of inferring support from an application name.

## Skill routing and prepared launch

- **Three authoritative skills.** Use `amq-squad:wizard` to prepare a team and
  accepted goal, `amq-squad:cli` for direct status and lifecycle operations,
  and `amq-squad:orchestrator` only from the verified live lead. The former
  `amq-squad`, `amq-squad-orchestrator`, `amq-team-setup`, and
  `amq-squad-role-creator` skill names remain compatibility redirects rather
  than independent instruction sources.
- **Goal-first, two-approval launch.** Project runs now follow a read-only
  proposal, a default-No preparation approval, exact readiness validation, and
  a separate default-No launch approval. Preparation persists the accepted
  generation, launch shape, goal source, digest, and goal binding; the live
  launch must consume that exact prepared identity instead of reconstructing
  it from mutable inputs.
- **Pinned transactional generation.** Preparation, readiness, launch-record
  persistence, roster changes, and goal reservation are checked against the
  same prepared generation. Compare-and-swap checks fail closed when an
  external writer changes the record, and rollback only removes state still
  owned by the failed attempt.
- **Claim-once goal delivery.** The lead's accepted goal is reserved before
  launch and delivered once to the verified lead identity. Delivery evidence
  remains internal to the managed launch contract; workers do not receive the
  lead goal binding or its prepared token.

## Managed resume and lead readiness

- **Exact managed replay.** Resume validates the persisted prepared record,
  raw record snapshot, semantic launch identity, delivered lead binding, and
  delivery receipt before it emits or executes a restore command. The replay
  command carries the exact restored prepared identity, while the child
  boundary strips internal token and descriptor inputs after validation.
- **Reattach versus re-orient.** A member with a saved conversation reattaches
  and keeps bootstrap suppressed. A restorable member without a saved
  conversation re-orients: bootstrap runs again so the agent rereads its role
  and brief and drains durable AMQ context.
- **Lead-first readiness.** Managed resume starts and checks the configured
  lead before launching dependent roles. A missing, mismatched, or unready lead
  blocks those dependents instead of producing a partially coordinated team.
- **Same-CWD identity.** Prepared and resumed members retain the canonical
  target project root as both launch identity and process working directory,
  including external-lead and split caller/child working-directory paths.
  Equivalent platform path aliases are normalized before identity comparison.

## Terminal capability policy

Terminal integration is capability-specific. Managed tmux panes provide the
full launch, visibility, focus, input, dispatch, and lifecycle contract. Native
terminal backends advertise only operations proven by their runtime context;
an application label or generic terminal environment does not imply focus,
prompt injection, stop, or resume support. In particular, iTerm2-specific
control is not attributed to a generic VS Code terminal. Unsupported or
unverified operations remain disabled and surface explicit reasons.

## Compatibility and release validation

- Go 1.25+ remains required.
- The minimum AMQ floor remains 0.42.1. This release is explicitly validated
  against pinned 0.43.1; latest remains a forward-compatibility canary.
- Claude and Codex plugin manifests, the canonical source skills, generated
  mirrors, README install example, HTML mirrors, and startup markers remain
  versioned together at v2.22.0.
- `make release-check VERSION=v2.22.0` now requires this canonical
  `docs/v2.22.0-release-notes.md` artifact and its exact release heading.

## Known non-blocking follow-ups

These items are intentionally documented rather than changed by the v2.22.0
ship-candidate notes commit:

- **A1 — degraded goal-delivery recovery.** If an unconfirmed pane submission
  falls back successfully to durable AMQ and the lead claims it, the binding
  can remain `reserved`. Claim-once is preserved and managed resume correctly
  fails closed, but the operator still needs documented guidance to create a
  fresh accepted launch; a later improvement may reconcile `reserved` to
  `delivered` from exact claim and receipt evidence.
- **A2 — stale resume-command recovery.** A printed resume command correctly
  fails when its prepared record snapshot loses the compare-and-swap race. The
  operator guidance should explicitly say to rerun `amq-squad resume` and use a
  newly emitted command from the current record.
- **C1 — identity-drift diagnostics wording.** Fresh launches scrub an empty
  injected `AM_SESSION` at the child boundary, but winning context rows can
  still contain contradictory "losing candidate" wording. The related
  stop-and-relaunch guidance should also identify the operator, rather than the
  agent shell, as the actor who can perform that recovery.

## Release boundary

This document prepares the v2.22.0 ship candidate. It does not create or push a
tag, merge a branch, publish a GitHub release, or otherwise perform a remote
lifecycle action; each remains a separately verified operator action.

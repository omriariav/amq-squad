# amq-squad v2.11.0 Plan

## Theme

v2.11.0 is the runtime-invariant release for the global-orchestrator and
project-lead model.

v2.10.0 established the vocabulary and enough runtime metadata to describe the
model: execution modes, visible leads, goal binding, namespace-safe status,
operator delivery, and delivery receipts. v2.11.0 must make the model hard to
misuse.

The release goal is:

> Reliable lead operation without manual pane watching, accidental role collapse,
> or lead-boundary bypass.

## Intended Operating Model

- A root/global orchestrator can run from a stable root such as `~/Code`, or
  through amq-noc.
- Each project is controlled through either:
  - one visible project lead with a native `/goal`; or
  - one visible project team with a lead.
- The project lead owns implementation, review coordination, validation, release
  readiness, and final operator-facing reports.
- The global orchestrator monitors, routes operator directives, answers gates,
  and tracks progress. It does not silently become the mutable project actor.
- Child agents, reviewers, implementers, and specialists are lead-internal unless
  escalated by the lead or surfaced as read-only evidence.
- amq-squad owns the protocol and runtime contracts.
- amq-noc owns the operator cockpit and consumes amq-squad JSON/action
  contracts.

## Current State After v2.10.0

The code has useful foundations:

- team config enforces one configured lead on orchestrated teams;
- `team init` persists execution-mode metadata;
- `status --json` emits execution, operator delivery, goal binding, topology,
  records, and runtime actions;
- `global_orchestrator` is described as `implementation_allowed=false`;
- virtual operator handles correctly report `poll_required=true`;
- operator delivery metadata exists, but the operator is still partly virtual
  rather than a first-class non-runnable participant;
- managed tmux launches record runtime identity;
- `lead register` explicitly adopts the current pane as an external lead;
- prompt delivery tries to confirm Enter submission and can return
  `SubmitUnconfirmedError`.

The problem is that the product boundary is still too advisory:

- "global orchestrator must not mutate project code" is metadata and prose, not
  a runtime guard;
- "project lead owns implementation/review/release" is team-rule and skill
  guidance, not a CLI invariant;
- "children are hidden/internal unless escalated" is not enforced by runtime
  actions;
- "lead is live" is not separated from "lead is operator-visible, attached, and
  repairable if detached";
- `user`/operator can still be confused with runnable workers in some control
  surfaces instead of being mailbox-only;
- sibling-window launch from a global/root orchestrator has no explicit
  preflight or repair contract when it falls back to a managed tmux session;
- `amq drain`, `amq watch`, `amq read`, and `collect` can consume another
  actor's mailbox without a project-lead boundary check;
- `goal deliver` can target a resolvable live role pane without proving it is a
  valid operator-visible lead;
- `live` can be reported even when the pane is not a valid distinct visible lead
  pane.

## Root Causes To Fix

### Visible lead identity is under-specified

`status --json` currently treats launch record plus liveness as enough. It needs
to distinguish:

- live process;
- live pane;
- operator-visible lead;
- attached operator surface;
- managed visible pane/window;
- detached but repairable managed session;
- explicit direct-lead adoption;
- invalid same-pane/current-pane collapse.

Issue #258 adds the launch-topology half of this invariant: a global/root
orchestrator must get a machine-readable topology preflight before mutation, and
status must expose whether a live lead is actually operator-visible or merely
live in a detached or managed tmux surface.

### Operator-visible topology is implicit

The runtime needs a first-class contract for the operator surface:

- whether the launcher is inside tmux;
- the current tmux session/window/pane when detectable;
- whether sibling-window launch is possible from that surface;
- whether a fallback managed tmux session was used;
- whether that fallback is attached, detached, or repairable;
- how to launch a missing role into an existing managed team session.

`--target new-window` and managed tmux fallback must not degrade into generic
"session already exists" failures. If a project-team session already exists,
launching another role should add a managed window there or return a specific
repairable topology error.

### Execution mode is descriptive, not enforced

Execution mode can say `global_orchestrator` or `project_lead`, but commands do
not consistently enforce who may mutate project code, who may own `/goal`, or
which pane is allowed to count as the visible lead.

### Goal binding can be degraded without blocking project-lead work

`native_goal_missing` is visible in status, but project-lead execution can still
continue with weaker binding. v2.11 should make missing native goal binding a
clear invariant failure or explicit fallback mode with visible risk.

### Wake/nudge receipts are too coarse

Dispatch can write durable AMQ successfully while pane delivery is only partially
successful. Current receipt statuses collapse important states such as prompt
staged, Enter submitted, submit unconfirmed, drain observed, and drain timed out.

### Lead-owned mailboxes are unguarded

The project-lead flow requires reviewer/child reports to go child -> lead ->
operator. The AMQ plumbing still allows a global orchestrator or operator client
to drain child/lead-owned inboxes directly.

### Operator polling is ad hoc

`operator_delivery.poll_required=true` says polling is needed, but there is no
poll owner, lease, cursor, backlog, freshness, or duplicate-poller warning.

### Operator participant is not first-class

Issue #259 makes the human operator an explicit team/profile participant without
turning the operator into a worker. New profiles should include an operator
contract such as `handle=user`, `kind=operator`, `runnable=false`,
`assignable=false`, `wake_supported=false`, and `poll_required=true`.

This is the schema/runtime foundation for operator delivery (#250), lead-owned
mailbox boundaries (#254), operator-visible topology (#258), and wake/nudge
state (#255). amq-noc and global orchestrators must discover the canonical
operator inbox and poll/peek actions from amq-squad JSON instead of
reconstructing AMQ paths.

### Delegation is advisory

Leads can solo complex release work without persisted delegation posture,
review-plan state, or release-readiness gates.

## 2.11.0 Issues

Milestone source of truth:

- #256 + #258: visible-lead/operator-surface invariant. v2.10.0 can collapse
  orchestrator and visible lead into the same/hidden Codex pane while reporting
  `visible_lead` as live, and it lacks explicit, repairable topology state for
  sibling-window launch, detached managed sessions, and existing managed team
  sessions.
- #255: Make pane wake/nudge submit reliable and receipt-backed.
- #254: Enforce lead-owned coordination boundaries for child reviews and
  operator reports.
- #251: Define lead delegation and resource-use rubric.
- #250: Design operator delivery loop: wake, polling, NOC, or bridge.
- #257: Make runtime action metadata policy-aware in project-lead mode.
- #259: Promote the operator to a first-class non-runnable participant with a
  canonical inbox/poll contract.

## Recommended Execution Order

1. Fix visible-lead identity and operator-surface topology invariants (#256,
   #258).
2. Make runtime actions policy-aware (#257).
3. Add the operator participant contract and worker-operation refusals (#259).
4. Expand wake/nudge delivery state (#255).
5. Add lead-owned mailbox guards and read-only inspection (#254).
6. Add operator loop status/poll/watch contracts (#250).
7. Persist delegation/review posture and release-readiness gates (#251).

This ordering matters. If the visible lead identity is false, operator-loop and
NOC state are built on the wrong actor.

## Must

### Launch and visibility invariants

Persist enough launch/adoption metadata to prove how a lead became visible:

- `execution_mode`
- `adoption_mode`
- `operator_surface`
- `launcher_pane_id`
- `agent_pane_id`
- `attached_client_count`
- `operator_visible`
- `visibility_error`
- `visibility_repair_actions`
- `managed_target`

Hard preflights:

- Refuse an orchestrated project-lead visible lead launched by manual `agent up`
  from a tmux pane with no managed target, unless it is explicit
  `direct_lead_session` or `lead register`.
- Refuse or mark invalid when `launcher_pane_id == agent_pane_id` and the mode
  was not explicit direct-lead adoption.
- Preflight visible launch from a global/root orchestrator before mutation:
  `TMUX` presence, current tmux identity, sibling-window capability, and fallback
  target.
- If sibling-window launch falls back to a managed session, record that topology
  and expose an attach/open repair action instead of treating `live` as
  operator-visible.
- If a managed team tmux session already exists, launching another role into the
  same project team must add a managed window there or return a specific
  repairable error.
- Make `goal draft` visible-lead happy path use a managed visible launch or an
  explicit registration path, not a bare ambiguous `agent up`.

Status fields:

- `records[].operator_visible`
- `records[].adoption_mode`
- `records[].role_boundary`
- `records[].current_pane_conflict`
- `records[].attached_client_count`
- `records[].visibility_problem`
- `records[].visibility_repair_actions[]`
- `execution.invariant_ok`
- `execution.invariant_errors[]`

### Native goal binding guard

For project-lead mode:

- `goal deliver` should target only the configured visible lead by default.
- It should refuse non-visible/current-pane-collapsed targets unless an explicit
  direct-lead override exists.
- `status --json` should separate `native_goal_missing` from valid fallback
  modes and make degraded goal binding actionable.

### Runtime action authorization

`status --json.actions` must be policy-aware because amq-noc renders those
actions directly.

In `project_lead` mode:

- global/operator actions should route directives through the visible lead;
- child `send`, `goal_deliver`, and drain-like actions should be hidden,
  unavailable, or marked override-only by default;
- read-only inspect/peek actions should be available;
- unsafe overrides should require confirmation, reason, and audit.

### Operator participant guardrails

New team/profile creation should persist an explicit operator block by default:

- `enabled=true`
- `handle=user` unless configured otherwise
- `participant=true`
- `kind=operator`
- `runnable=false`
- `assignable=false`
- `wake_supported=false`
- `poll_required=true`

Compatibility:

- Existing profiles without an explicit operator block remain valid and keep
  today's implicit operator behavior.
- `--no-operator` remains an explicit opt-out.
- `user` must not be inserted into normal runnable `members[]`.
- If AMQ lacks native team participant support, status should expose
  `amq_team_participant=false` while preserving amq-squad's virtual operator
  contract.

Runtime must refuse treating the operator as a worker:

- no `agent up user`;
- no `team member add user` when `user` is the operator handle;
- no task/dispatch assignment to `user`;
- no `goal deliver --role user`;
- no `focus`, `send`, `resume`, or `stop` for the operator handle;
- no `team lead set user`;
- self-send on operator p2p threads fails by default.

Status should expose the operator separately from runnable members:

- `operator.enabled`
- `operator.handle`
- `operator.kind`
- `operator.runnable=false`
- `operator.assignable=false`
- `operator.canonical_inbox`
- `operator.poll.required`
- `operator.poll.owner`
- `operator.poll.cursor`
- `operator.poll.unread`
- `operator.poll.open_gates`
- operator-scoped actions such as `operator_poll`, `operator_peek`,
  `operator_reply`, `operator_directive_to_lead`, `gate_answer`, and
  `gate_inspect`.

### Delivery state machine

Represent pane/nudge delivery with explicit stages:

```text
queued_amq
  -> nudge_requested
  -> prompt_staged
  -> submit_attempted
  -> submit_confirmed | submit_unconfirmed | submit_failed
  -> drain_observed
  -> drain_acked | drain_timeout
```

Safe retry must not duplicate the durable task body. It should only re-send the
fixed drain-only nudge.

### Lead-owned mailbox boundaries

Add read-only inspection for global/NOC use:

- inspect/peek inbox;
- inspect/peek thread;
- inspect child evidence without moving receipts.

Add guarded mutation:

- default child/reviewer inbox drains are blocked in project-lead mode;
- overrides require `--override-boundary` and `--reason`;
- overrides write an audit record;
- recovery/requeue guidance is available when a lead-owned message was consumed
  by the wrong actor.

### Operator loop contract

Add the product contract before building a daemon. This consumes the #259
operator participant/canonical inbox contract rather than inventing independent
AMQ path reconstruction:

- `operator status --json`
- `operator poll --json --readonly`
- `operator watch` as the reference CLI loop
- poll lease and owner conflict
- cursor/backlog/freshness fields

Suggested state:

```text
unconfigured
  -> poll_required_unowned
  -> poller_active
  -> poller_stale
  -> offline_recovery
```

Suggested JSON:

```json
{
  "operator_loop": {
    "mode": "poll",
    "poll_required": true,
    "owner": "none|cli|noc|daemon",
    "owner_id": "cli:<host>:<pid>",
    "lease_expires_at": "...",
    "last_poll_at": "...",
    "cursor": "...",
    "backlog": 0,
    "gates_open": 0,
    "directives_unacked": 0,
    "degraded_reason": ""
  }
}
```

- Add automatic remediation hints such as:
  - "register current pane as direct lead";
  - "launch visible lead in new window";
  - "attach managed visible session";

## amq-noc Boundary

amq-squad owns protocol and runtime semantics:

- namespace identity;
- visible lead identity;
- operator-visible topology and repair contracts;
- goal binding;
- runtime action policy;
- operator-loop semantics;
- delivery health;
- lead-owned mailbox boundaries.

amq-noc owns cockpit UX:

- refresh cadence;
- loop owner/freshness display;
- lead-centered project view;
- visible topology display and repair actions;
- gate/directive conversation flow;
- delivery ladder and stuck-nudge alerts;
- hiding or demoting unsafe child actions.

NOC may poll for display. amq-squad defines what polling means.

## Tests Required

- Manual `agent up` from the current/root pane must not become a valid visible
  project lead.
- Explicit `lead register` / direct-lead session remains valid and visibly marked.
- `goal deliver` refuses non-visible/current-pane-collapsed targets by default.
- `status --json` reports `operator_visible=false` and invariant errors for bad
  visible lead state.
- Visible-team launch preflight reports `TMUX` presence, current tmux identity,
  sibling-window capability, and fallback target before mutation.
- `status --json` separates `live`, `pane_alive`, `operator_visible`,
  `attached_client_count`, `visibility_problem`, and
  `visibility_repair_actions`.
- A live but detached managed lead reports `operator_visible=false` plus an
  attach/open repair action.
- Existing managed team session plus missing visible role launches a new managed
  window or returns a specific repairable topology error.
- Runtime action JSON hides or marks direct child actions override-only in
  project-lead mode.
- New teams/profiles persist an explicit non-runnable operator participant by
  default, while legacy profiles retain implicit operator compatibility.
- `status --json` exposes the canonical operator inbox and poll state
  separately from runnable members.
- Worker lifecycle/control/task/goal/lead operations refuse the operator handle
  with a clear mailbox-only error.
- Dispatch exposes `submit_unconfirmed` distinctly.
- Queued AMQ plus staged prompt but no Enter creates a stuck delivery warning and
  a safe re-nudge action.
- Global/operator drain of child/reviewer inbox is blocked or audit-gated.
- Read-only inspect/peek does not move messages to current/drained state.
- Operator poll lease conflict is deterministic.
- Release-ready without lead final recommendation plus review/waiver fails.

## Dogfood Plan

Run v2.11.0 using the target model:

- root/global orchestrator from `~/Code` or amq-noc;
- one managed visible release lead in a distinct operator-visible pane/window;
- native `/goal` verified before implementation;
- release lead declares `solo`, `delegated`, or `visible_team` early;
- at least one independent reviewer unless explicitly waived;
- operator loop run through `operator watch` or amq-noc lease owner;
- no direct child/reviewer drain by the global orchestrator;
- deliberate stuck-nudge fixture or live smoke to verify warning/retry behavior;
- final release recommendation comes from the visible lead.

## Definition Of Done

- The unsafe states from v2.10 dogfood are impossible, refused, or explicitly
  marked invalid in JSON/TUI.
- amq-noc can consume amq-squad contracts without inventing polling, drain, or
  wake semantics.
- A root/global orchestrator can safely supervise project leads without becoming
  the project lead.
- Operator-visible lead identity, goal binding, delivery, mailbox ownership, and
  delegation posture are all machine-checkable.

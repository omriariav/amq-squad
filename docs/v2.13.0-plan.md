# amq-squad v2.13.0 Plan

## Theme

v2.13.0 is the robustness and skill-simplification release.

v2.12.0 delivered intent-first entry, canonical action contracts, and the
operator cookbook. v2.13.0 must harden the dispatch loop, surface stalled
native goals automatically, and reduce the skill maintenance surface by moving
toward one canonical skill source.

The release goal is:

> An orchestrated run should complete without manual `rm -rf` interventions,
> with blocked goals surfaced automatically, and with a single canonical skill
> source that requires no per-binary maintenance.

## Current State After v2.12.0

The code has:

- Named profile namespaces for AMQ roots, briefs, and task stores.
- Canonical action objects across status and runtime control surfaces.
- Operator delivery and poll-required metadata.
- Intent-first goal draft/start surfaces.
- Repair-first status metadata for visible lead and topology failures.
- A root operator cookbook and v2.12.0 handoff digest.

The remaining release gap:

- A named-profile launch can still recreate the legacy `.agent-mail/<session>`
  root when bootstrap or `agent up` resolves by `--session` instead of the
  profile root.
- Presence-only bootstrap residue under a legacy root is treated as durable
  state, blocking resume/dispatch until an operator manually deletes it.
- A lead with a blocked native `/goal` is not surfaced as an actionable
  orchestrator blocker.
- Claude and Codex skill files still duplicate most instructions and can drift.
- The release needs a fresh handoff digest before tagging.

### Legacy AMQ Root Fix

Named-profile launches must never write bootstrap, presence, launch records, or
messages to `.agent-mail/<session>` when the profile root is
`.agent-mail/<profile>/<session>`.

Required behavior:

- `agent up --team-profile PROFILE --session SESSION` derives the profile AMQ
  root even if the caller omitted `--root`.
- Generated team launch commands continue to pass the profile root explicitly.
- Conflict detection ignores presence-only residue in a legacy root.
- Conflict detection still blocks when the legacy root contains real durable
  state such as messages, launch records, or non-empty agent extension data.

Acceptance: a named-profile orchestrated run can dispatch without manual
`rm -rf .agent-mail/<session>` cleanup, and focused tests cover both prevention
and guard behavior.

### Blocked Goal Detection

When a visible lead's native `/goal` is blocked, `status --json` must expose a
machine-readable blocked-goal signal and the operator poll surface must count or
list it as an actionable blocker.

Acceptance: tests prove `native_goal_blocked` or equivalent status metadata is
emitted, and the orchestrator skill documents detection, surface, gate, and
`/goal resume` flow.

### Skill Simplification

Reduce skill drift between Claude and Codex plugin variants.

Required behavior for this release:

- `goal draft --skill-invocation` emits a ready-to-paste orchestrator
  invocation block.
- Claude and Codex plugin skill mirrors are generated from canonical
  `plugins/skills-src/*/SKILL.md` sources with per-binary frontmatter.
- The primary skill surface is two skills: `amq-squad` for setup, role
  authoring, and live coordination; `amq-squad-orchestrator` for lead-agent
  bootstrap.
- `amq-team-setup` and `amq-squad-role-creator` remain as generated redirect
  stubs, not deleted.

Acceptance: the drafted invocation path is covered by tests, `make
skills-check` verifies generated mirrors are fresh, and the plugin outputs show
the 4-skills-to-2-skills decision with redirect stubs.

### External Lead Wake Registration

`lead register` must not leave an externally registered orchestrator pane as a
poll-only island when AMQ wake support is available.

Required behavior:

- `lead register` starts or repairs `amq wake` for the resolved external lead
  handle/root by default.
- `--wake` makes the default wake startup explicit.
- `--no-wake` skips wake startup explicitly.
- `--no-require-wake` records a non-fatal wake failure and keeps registration
  usable for polling/manual collect.
- Wake injector flags are accepted and persisted on the external launch record.

Acceptance: focused tests cover explicit/default wake startup, skipped wake,
non-required wake failure, injector persistence, conflict validation, and
cleanup of failed wake processes.

### Resume Exec Stale-Record Recovery

`resume --exec` must not fail with `stale-record` when tmux already launched
the requested pane but the agent is still bootstrapping and has not refreshed
`launch.json` before the verification timeout.

Required behavior:

- After the verification timeout, scan live tmux panes for the deterministic
  `amq:<session>:<role>` title and matching binary/cwd.
- If a matching pane exists, refresh the existing launch record's tmux identity
  and `StartedAt`, then re-run verification before returning an error.
- Do not adopt ambiguous panes without the deterministic role title.

Acceptance: focused tests cover stale-record adoption by pane title and preserve
the existing partial-failure behavior when no matching pane exists.

### Goal Delivery Orchestrator Registration

`goal start` and `goal deliver` should support a non-agent/global-orchestrator
launcher pane becoming observable before it delivers the native goal.

Required behavior:

- Add `--register-orchestrator[=HANDLE]` to `goal start` and `goal deliver`;
  the default handle is `orchestrator`.
- Before non-dry-run delivery, ensure the team has an `orchestrator` member,
  write an external launch record for the current pane, start/repair wake, and
  mark the profile lead as `orchestrator`.
- Preserve explicit `--role` delivery targets so registering the launcher does
  not accidentally retarget the native goal.

Acceptance: tests cover default/valued optional flag parsing and JSON goal
delivery with orchestrator registration, wake startup, team mutation, and a
clean delivery envelope.

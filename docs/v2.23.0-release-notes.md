# amq-squad v2.23.0

## Summary

v2.23.0 closes milestone #30 by making launch generations, task lifecycle
events, and actor capabilities exact durable contracts. One immutable prepared
generation now survives preparation, launch, staged spawn, rollback, goal
admission, and managed resume. Task completion requires structured terminal
evidence correlated to that generation. Bootstrap prompts describe the current
actor's implementation and delegation capability instead of inheriting the
planner lead's posture.

## Immutable prepared generations (#485)

- `run start --prepare` publishes an immutable schema-3 generation manifest and
  initial state. The session current pointer names both artifacts by digest;
  older generations remain immutable and addressable.
- Generation IDs and launch/resume attempts are canonical lowercase 32-hex path
  tokens. The generation reference binds generation, manifest digest, goal
  namespace, and goal digest everywhere it crosses a process or record boundary.
- Launch reservation, initial-member claims, the accepted goal, staged-child
  reservation/claim, terminal state, and resume attempts are append-only,
  claim-once artifacts. Replay cannot make an accepted generation reusable.
- Goal admission holds the profile, actor-record, and prepared-generation lock
  domains through the final identity checks, attempt publication, launch-binding
  compare-and-swap, and goal claim. Either a concurrent mutation wins with no
  side effects, or admission wins as one complete durable transaction.
- Staged actors must be complete configured members at preparation time. The
  generated `agent up --staged-spawn` path verifies identity, binary/model/args,
  tool policy, actor mode, bootstrap digest, current pointer, and completed
  initial launch before persisting one child attempt.
- Crash-safe same-directory publication, no-replace links, directory sync, and
  exact rollback ownership prevent partial artifacts and replacement races.

## Structured task and AMQ lifecycle correlation (#487)

- Task-backed dispatch binds task ID, actor, profile/session namespace, complete
  prepared-generation reference, claim generation, dispatch message, outbox
  intent, lifecycle event ID, and immutable evidence where the event requires it.
- ACK, PROGRESS, CHECKPOINT, and REVIEW are explicit nonterminal events. DONE,
  BLOCK, and CANCEL are committed by terminal task commands in the same
  transaction as task state, journal, and outbox changes.
- Subject heuristics and arbitrary prose cannot satisfy completion. Duplicate,
  delayed, stale-generation, wrong-actor, wrong-anchor, or cross-namespace
  traffic fails closed and remains available for diagnosis.
- Event ordering follows the committed append-only task journal, not a sender's
  `occurred_at` timestamp. Exact value equality is used for evidence and
  generation references across JSON round trips.
- The retrospective t44 ACK message shape is a permanent regression fixture and
  cannot trigger a completion warning or state transition.

## Actor-relative bootstrap contracts (#492)

- Team schema 5 stores an exact `actor_mode` (`implementation` or `review`) for
  every member. Schema-5 omissions fail closed; schema-1 through schema-4 retain
  their documented historical effective behavior.
- Rendered bootstrap contracts carry actor role, actor handle, lead status,
  implementation permission for this actor, delegation permission for this
  actor, and team lead mode.
- Planner leads remain review-only and delegate implementation. Implementation
  workers execute their own bounded assignments directly and do not inherit the
  lead's no-implementation posture or delegation authority.
- Role and handle lookup is trimmed but case-sensitive across bootstrap,
  prepared manifests, launch records, staged claims, and goal admission.

## Integration and compatibility

- The prepared-token `LaunchAttempt` is copied directly into launch records and
  delivery receipts, including structured task outbox receipts; no reflective
  compatibility path can silently drop it.
- Runtime lifecycle fixtures use canonical generation IDs and publish through
  the generation-addressed prepared-state helpers.
- Completion metadata includes the new `--actor-mode` and `--staged-spawn`
  surfaces.
- Go 1.25+ remains required.
- The minimum AMQ 0.42.1 compatibility floor is unchanged. This release is
  explicitly validated against pinned 0.43.1; latest remains a
  forward-compatibility canary.

## Migration and recovery

Older preparation files are not upgraded in place. Preserve their evidence,
run the current binary's read-only proposal, and create a fresh reviewed
preparation. Do not delete generation claims or replay saved launch/resume
commands. For a staged child, use only the generated `agent up --staged-spawn`
command after the initial generation has exact completion evidence.

Task records and AMQ prose do not acquire structured lifecycle authority during
upgrade. Use task-backed dispatch from launch records that match the current
accepted prepared generation. See
[the v2.23.0 migration guide](v2.23.0-platform-migration.md) for exact recovery
guidance.

## Release boundary

This document prepares the v2.23.0 ship candidate. It does not merge a branch,
push a commit or tag, publish a GitHub release, close issues or milestone #30,
or authorize any other remote lifecycle action. Each action remains behind its
own exact operator approval and verification.

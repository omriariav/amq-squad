## The shift

Today you **design a team, then run it**: `team init` authors a static roster,
`up` spawns exactly that roster, composition is frozen for the session. 2.0
inverts this: **you give a lead a goal, and the lead composes the team** ŌĆö
proposing, spawning, and pruning agents as the work reveals what it needs.

The load-bearing constraint, and the reason this is amq-squad-native rather than
"just use Claude Code Agent Teams + the `amq swarm` bridge": **orchestration must
be binary-neutral ŌĆö a Codex agent must be able to *lead*, not just be led.** Agent
Teams' composition authority (task creation, the `~/.claude/...` substrate) is
Claude-Code-bound; a Codex agent can only join as a worker. amq-squad keeps the
orchestrator binary-agnostic.

### The composition spectrum (manual is the floor)

| Mode | Who composes | Status in 2.0 |
| --- | --- | --- |
| **Manual** | Human designs the roster up front (today's `team init`/wizard). | first-class, unchanged |
| **Seeded** | Lead **proposes** spawns from the goal; **operator approves each**. | **2.0 default** |
| **Autonomous** | Lead spawns/removes within guardrails, no per-spawn approval. | **deferred to 2.1** |

2.0 ships **manual + seeded only.** Autonomous mode waits for 2.1 ŌĆö after the
guardrails and telemetry that make unsupervised spawning safe are proven. "Goal-
first" changes the default *mental model*; "seeded" keeps the default *safety
posture* propose-then-approve.

## The bundle

1. **Headline (new):** goal-first dynamic composition ŌĆö a binary-neutral lead that
   assembles and adjusts its team from the goal.
2. **Housekeeping (deferred debt):** remove the `down`/`launch`/`restore`/`list`/
   `team show`/`team launch` stubs, and do the `/v2` module rename.
   *(The `Team.Workstream` pin shim stays for 2.1 ŌĆö its own code note says
   "removal in 2.1"; 2.0 leaves it deprecated.)*

## Architecture

Everything new is **binary-neutral** and amq-squad-native (no `~/.claude/`
dependency in the core), so a Codex lead is first-class. We borrow the *mechanics*
proven by `amq swarm` but keep the data and authority in `.amq-squad/`.

### 1. Mutable roster (`team member add/rm`)

`team.json` becomes a runtime artifact.

- New `team member` subcommand (a `case "member"` in `runTeam`): `team member add
  <role> --binary <claude|codex> [--model M] [--claude-args ŌĆ”]` (the `--up`/`--stop` auto-launch flags are deferred to Phase 1) and `team member rm <role>`.
- **Atomicity:** reuse the existing atomic write (`team.WriteProfile` ŌåÆ write
  `.tmp` ŌåÆ `os.Rename`); add the swarm-style **exclusive file lock** on a sidecar
  `.lock` so concurrent adds from lead + workers can't interleave. All mutation
  paths go through `WriteProfile` (never a direct write), so a crash mid-add
  leaves valid JSON.
- **Empty-roster guard:** today `len(Members)==0` is rejected in ~7 sites
  (`up.go:193`, `team_launch.go:155`, `team_resume.go`, `down.go:243`,
  `status.go:169`, `fork.go:80`, `bootstrap.go:188`). 2.0 allows a **lead-only
  seed**; in 1.x this is gated **off by default** (see Backward-compat).

### 2. Native pull-based task model (`.amq-squad/tasks/<session>/`)

A new `internal/task` package + verbs make dispatch **task-pull**, not only
pane-push:

- `task add --title ŌĆ” [--depends-on ID,ŌĆ”] [--assign ROLE]` ŌĆö **the lead
  decomposes the goal into tasks.** (Unlike `amq swarm`, which has *no create
  verb* and leans on a Claude lead's native `TaskCreate`, our store has `add`, so
  an any-binary lead can decompose.)
- `task list [--status ŌĆ”]`, `task claim <id>`, `task done|fail|block <id>` ŌĆö five-
  state machine (`pending ŌåÆ in_progress ŌåÆ completed|failed|blocked`), **dependency
  gating** (can't claim until `depends_on` are `completed`), exclusive file-lock +
  atomic-rename writes (mirror `swarm` task semantics, which were verified).
- A short `docs/task-store-design.md` (state diagram, schema, atomic protocol,
  crash/partial-write recovery) precedes the implementation.

### 3. Seeded approval gate ŌĆö the `gate/<topic>` pattern (not #117 directives)

**Correction from review:** #117 directives are operatorŌåÆlead *steering* (`--kind
todo` on the p2p thread) ŌĆö orthogonal to approval. The existing **human-approval
channel is `gate/<topic>`** with `question`/`answer`. Seeded spawn approval reuses
exactly that:

1. Lead proposes a spawn ŌåÆ emits `amq send --to <operator> --thread
   gate/spawn-<role> --kind question --subject "APPROVAL: spawn <role> ŌĆö <reason>"`.
2. Lead **blocks** on `amq drain --thread gate/spawn-<role>` (bounded timeout).
3. Operator answers `--kind answer` APPROVED/DENIED (from the NOC or CLI).
4. **Fail-safe:** timeout or DENIED ŌćÆ do not spawn; the lead must re-propose.

A `docs/seeded-approval-flow.md` specifies the message shapes, blocking/timeout,
and the operator's NOC-vs-CLI paths before this is coded.

### 4. Durable dynamic roster ŌåÆ resume rebuilds the team

Today `team resume` only restores `team.json` members; ad-hoc `agent up` children
fall out (verified). Because the lead now *builds* the roster, its adds persist to
`team.json`, so `resume` rebuilds the team the lead assembled. A
**spawn-decision log** (`.amq-squad/sessions/<session>/spawn-plan.jsonl`, append-
only: proposed role / reason / status / approval) gives the operator an audit
trail and lets `resume` be **idempotent** ŌĆö re-bring only what isn't already live,
never double-spawn. A `team validate --session S [--fix]` verb checks
rosterŌåödirectory coherence before launch.

### 5. Guardrails (seeded now; the rest gate autonomous in 2.1)

- **Caps now:** `--max-agents` (config field + flag), checked before every `team
  member add` ŌĆö even seeded mode shouldn't sprawl.
- **For 2.1 / autonomous:** budget ceiling, idle reaping, recursive-spawn
  prevention, and prompt-injection review of the lead's spawn heuristics (no
  dynamically-generated role names from raw goal text).

### 6. Optional Agent-Teams interop (edge adapter, deferred to 2.1)

In 2.0 **all squads ŌĆö Claude- and Codex-led ŌĆö use amq-squad-native tasks.** A
later optional adapter can mirror `.amq-squad/tasks/` Ōåö `~/.claude/tasks/` for a
Claude-led team that wants to surface to native Agent Teams; it is never a core
dependency (that would re-bind orchestration to Claude). Codex-led teams stay
amq-squad-native.

### 7. The orchestrator skill: "compose from the goal"

The hard 80% is **agent judgment**, not the CLI. The `amq-squad-orchestrator`
skill (both mirrors) gains a composition playbook:

- A **role library** the lead draws from: the built-in catalog (shipped in the
  binary, surfaced by `amq-squad roles`) plus staged custom roles under
  `.amq-squad/roles/`. `team member add` validates the role exists/non-empty.
- Heuristics: read the goal ŌåÆ propose a minimal team ŌåÆ (seeded) request approval ŌåÆ
  spawn ŌåÆ decompose into tasks ŌåÆ monitor ŌåÆ **prune** idle/done agents ŌåÆ grow when
  blocked. Worked examples in the skill doc.
- Anti-patterns: over-spawning (cost), under-spawning (serialization), orphaning,
  treating a child report as authority.

## Backward-compatibility (honest framing)

Phase 0 is **additive in surface** (no removed/renamed verbs, no forced schema
break) but introduces **two intentional behavior changes**, both **gated off by
default in 1.x** so existing workflows are untouched:

- lead-only seed (relaxed empty-roster guard), and
- `resume` respecting the dynamically-built roster.

The gate is an opt-in marker (an additive `team.json` field with `omitempty`, e.g.
`"dynamic": true`, **schema stays v3-loadable** ŌĆö no migration needed for old
files). Phase 1 (2.0) flips the default on. Release notes list the behavior
changes and the one-line migration ("if a script asserts `team.json` has members,
allow empty rosters"). A `make ci` drift-guard fails if the two skill mirrors
diverge.

## Sequencing ŌĆö de-risk the major

**The plumbing is proven; the agent judgment is not.** Validate that additively in
1.x, then cut 2.0 as a confident reframing + cleanup. The slices are a **critical
path, not parallel: A ŌåÆ B ŌåÆ C ŌåÆ D.**

### Phase 0 ŌĆö additive, opt-in, in 1.x (behavior changes gated off by default)

- **Slice A ŌĆö mutable roster.** `team member add/rm` (atomic + file-locked +
  re-validated, via `WriteProfile`); the lead-only seed gated behind the opt-in.
  *Prereq for B and C.*
- **Slice B ŌĆö native task model.** `internal/task` + `.amq-squad/tasks/<session>/`
  + `task add/list/claim/done/fail/block`, dependency gating, atomic locking;
  `docs/task-store-design.md` first. *Prereq for C.*
- **Slice C ŌĆö orchestrate behavior (seeded).** The "compose from the goal"
  playbook + role library + the `gate/<topic>` propose-then-approve loop, both
  mirrors. *Prereq for D.*
- **Slice D ŌĆö DOGFOOD + EVAL GATE (blocking).** Two parts:
  1. **Eval rubric** (`docs/eval-strategy.md`): 5ŌĆō10 representative goals, each
     with a hand-authored reference team; a script spins up a **Codex-led** squad,
     captures the proposed roster, and scores it (role coverage, team size within
     a budget, no over/under-spawn). A passing bar gates the slice.
  2. **Live dogfood on `os-omri-pm`** (Codex lead, lead-only seed ŌĆö one lead member, which works today; a 0-member empty seed is a Phase-1 opt-in): proposes a team,
     gets approval, adds a worker mid-session, dispatches a dependency-gated task,
     prunes. **Do not start Phase 1 until both pass and the lead's *judgment* ŌĆö
     not just the verbs ŌĆö looks right.**

### Phase 1 ŌĆö the 2.0.0 cut (breaking) ŌĆö only after the Slice-D gate is green

- Flip the **default** to goal-first/seeded (manual stays first-class).
- Caps enforced; spawn-decision log + idempotent `resume`; `team validate`.
- **Breaking cleanup:** delete `launch`/`restore`/`list`/`team show`/`team launch`
  stubs and remove `down` ŌĆö each is an intentional break; ship `MIGRATION.md`
  (`down`ŌåÆ`stop --all`, `restore`ŌåÆ`resume`/`agent resume`, `list`ŌåÆ`status`/
  `history`, `team show`ŌåÆ`up --dry-run`, `team launch`ŌåÆ`up`). Emit deprecation
  warnings on these verbs in the last 1.x release before they're removed.
- **`/v2` module rename** ŌĆö **mandatory for Go v2 module resolution** (a real
  v2.0.0 tag requires the `/v2` suffix in the module path + every internal
  import: ~100 `.go` files, `go.mod`, README install, smoke tests). A **late,
  scripted slice** (find/replace + `go build`), run last before release so it
  never gates feature work.
- Migration guide + docs/skills resync (both mirrors) + plugin-manifest bump +
  marketplace refresh + GitHub release + `make release-smoke`.

## Risks & open questions

- **Agent judgment is unproven** ŌĆö Slice D (eval rubric + Codex dogfood) is the
  real gate; everything downstream waits on it. A regression suite is needed
  because model/prompt drift can silently change composition quality.
- **Cost/sprawl** ŌĆö caps land with seeded; autonomous (budget, idle reaping,
  recursion guard, injection review) is deliberately held to 2.1.
- **Schema** ŌĆö additive `omitempty` fields keep v3 loadable; only bump to v4 if a
  genuine incompatibility appears, with versioned read + fixtures.
- **Optional telemetry** (`.amq-squad/telemetry.jsonl`, opt-in) to measure
  approval/deny rates and composition quality ŌĆö informs whether autonomous is safe
  for 2.1.

## Acceptance (concrete, falsifiable)

Given the goal **"Build a React component library with tests, docs, and CI/CD"**
and an empty `team.json`, a **Codex** lead:

1. proposes a team (e.g. `[cto, frontend-dev, qa]`) in human-readable form;
2. requests approval per spawn on `gate/spawn-<role>`; operator answers APPROVED;
3. spawns the approved members within the cap;
4. creates Ōēź5 dependency-gated tasks via `task add`;
5. prunes an agent left idle past threshold;
6. on `stop`ŌåÆ`resume`, the **lead-built** team and open tasks are rebuilt
   idempotently (no double-spawn);

ŌĆ”all through stable amq-squad commands. Plus: manual `team init` still works; the
eval rubric passes its bar; deprecated verbs are gone with migration notes; the
module is `/v2`. Autonomous mode is explicitly **not** in scope for 2.0.
